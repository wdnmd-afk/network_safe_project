# 暴力破解修复说明

## 漏洞成因

漏洞版没有失败次数阈值。只要候选列表里包含正确口令，接口就会持续检查到命中为止。

## 本实验修复点

修复版在连续失败达到 3 次后：

- 标记 `thresholdExceeded` 为 `true`。
- 标记 `rateLimitApplied` 为 `true`。
- 返回 `brute-force-rate-limit-blocked`。
- 停止继续接受后续候选口令命中。

## 生产系统建议

真实业务中，暴力破解防护通常需要组合策略：

- 账号维度失败次数计数。
- IP、设备、会话维度风险评分。
- 指数退避或固定窗口节流。
- 短时锁定与解锁流程。
- 验证码或二次验证。
- MFA。
- 弱口令检测和密码策略。
- 异常登录审计、告警和用户通知。

## 日志要求

日志应记录判断过程和安全信号，但不得保存候选口令明文。推荐记录候选数量、失败次数、阈值、是否节流、请求路径、用户 ID 和 trace ID。

## 易错点

- 只在前端限制提交次数是不够的，后端必须独立执行阈值判断。
- 触发阈值后不能继续检查后续候选口令。
- 防护逻辑不能影响正常单次正确登录。
- 事件日志不能泄露密码候选值。
