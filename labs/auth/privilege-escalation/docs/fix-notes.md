# 权限提升修复说明

## 根因

后端把请求体中的 `requestedRole` 当作授权依据。攻击方可以修改客户端请求，把普通用户声明为 `admin`。

## 修复点

修复版只使用登录态中的服务端角色：

```text
currentUser.role
```

请求体中的 `requestedRole` 只用于教学观察，不参与授权决策。

如果普通用户请求管理操作：

- 返回 `403`
- 不执行虚拟管理操作
- 记录 `privilege-client-role-admin-blocked`
- 在日志中保留服务端角色、客户端声明角色和阻断原因

## 防御效果

- 普通操作不受影响。
- 普通用户声明 `admin` 后仍无法执行管理操作。
- 事件日志可以复盘权限提升尝试。

## 真实生产建议

- 统一封装服务端权限策略。
- 对敏感操作做二次授权和审计。
- 用服务端会话、数据库角色或权限服务作为可信来源。
- 不信任前端传来的角色、权限、菜单、按钮状态或路由状态。
- 对连续提权尝试建立告警。
