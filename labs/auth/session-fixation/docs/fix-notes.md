# 会话固定修复说明

## 根因

会话固定的根因是系统在认证状态变化后继续使用认证前的会话 ID。攻击者如果提前知道这个 ID，就可能在用户登录后复用同一个 ID。

## 本实验修复点

修复版在教学登录完成后执行以下策略：

1. 读取 `preLoginSessionId` 仅用于学习观察。
2. 不把客户端传入的 ID 作为登录后的教学会话 ID。
3. 由服务端生成新的教学会话 ID。
4. 在事件日志中记录轮换结果。

修复版预期信号：

```text
session-fixed-id-rotated
```

## 与漏洞版的差异

| 观察项 | 漏洞版 | 修复版 |
|---|---|---|
| 接受客户端预登录 ID | 是 | 否 |
| 登录后轮换 ID | 否 | 是 |
| 攻击样例信号 | `session-fixed-id-bound` | `session-fixed-id-rotated` |
| 业务结果 | 教学登录成功 | 教学登录成功 |
| 安全结果 | 攻击者已知 ID 被绑定 | 攻击者已知 ID 失效 |

## 生产系统建议

真实生产系统还需要：

- 登录成功后强制 session rotation。
- 旧 session 立即失效。
- Cookie 设置 `HttpOnly`、`Secure`、`SameSite`。
- 服务端统一管理会话生命周期。
- 对异常来源、重复会话、登录前后 ID 未变化等行为做审计。
- 结合 CSRF 防护和登录态刷新策略，避免会话相关风险叠加。

## 本项目安全边界

本实验不修改真实登录 token，不读取真实 Cookie，不保存真实会话密钥。所有会话 ID 都是教学样例。
