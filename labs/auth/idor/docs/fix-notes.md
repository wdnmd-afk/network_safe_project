# IDOR 修复说明

## 根因

后端只根据 `orderId` 读取资源，缺少对象级授权校验。前端隐藏入口、按钮置灰或只展示自己的订单列表，都不能替代后端校验。

## 修复点

修复版在后端读取对象后校验：

```text
order.ownerUserId === currentUser.id
```

如果归属不匹配：

- 返回 `403`
- 不返回订单详情
- 记录 `idor-cross-user-order-blocked`
- 在日志中保留对象归属摘要和阻断原因

## 防御效果

- 正常用户读取自己的订单不受影响。
- 替换为他人订单 ID 后被阻断。
- 数据库事件日志可以复盘越权读取尝试。

## 真实生产建议

- 把对象级授权封装到服务层或策略层。
- 对所有对象读取、更新、删除接口统一校验归属或权限。
- 对连续越权尝试建立审计告警。
- 不在错误响应中泄露过多对象存在性信息。
- 不把不可预测 ID 当作唯一防护。
