# 鱼叉式钓鱼修复说明

## 1. 防御目标

鱼叉式钓鱼的防御目标是建立可复盘的流程控制，让学习者看到针对性上下文不能替代正式核验。

当前处于 `in-progress`，已建立目录、文档、元数据入口和后端受控 `review` API。以下内容是当前 API 与后续页面实现必须遵守的防御方向。

## 2. 固定防御流程

后续修复版应围绕固定流程展示：

- 角色核验：重要请求不能只凭显示名、职位、熟悉语气或业务上下文判断。
- 可信通道二次确认：付款、权限、凭据、附件和流程例外必须通过可信通道确认。
- 审批链：财务、人事、供应商和工程权限变更必须绑定正式流程。
- 最小授权：临时访问必须有范围、有效期、审批记录和复核节点。
- 隔离与举报：可疑链接、附件或请求先隔离，再通过安全流程复盘。

## 3. 日志摘要建议

当前后端 API 已接入统一事件日志，只允许记录：

- 固定案例 key。
- 固定检查策略 key。
- 风险类别。
- 后端决策。
- 建议动作。
- 学习信号。

不得记录：

- 不得记录真实姓名、邮箱、手机号、公司、部门、职位或社交账号。
- 不得记录完整邮件正文、IM 对话、真实链接、真实附件或真实业务材料。
- 不得记录凭据、验证码、Cookie、token、付款信息或外部目标信息。

## 4. ready 前置条件

后续进入 `ready` 前必须满足 case-study ready 例外标准：

- 保持 `mode: "case-study"`。
- 漏洞版和修复版都有可访问学习入口。
- 若提供 API，API 只能使用固定虚构案例数据，不连接真实外部目标。
- 自动化验证至少覆盖页面差异、API 差异或只读一致性验证中的两类。
- 文档明确禁止真实投递、真实画像采集、凭据收集、模板生成、第三方平台调用和攻击脚本。
- `variants[].supportsAutomation` 不得因页面测试、API 测试或只读验证脚本存在而误标为攻击脚本自动化。

## 5. 当前边界

当前不提供前端页面、`verify.ts`、`exploit.py` 或任何投递能力。当前后端 API 只读取固定 `caseKey` 与 `verificationPolicyKey`，并且事件日志只保存安全摘要。
