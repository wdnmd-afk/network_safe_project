# 网络钓鱼识别修复说明

## 1. 防御目标

网络钓鱼识别的首版防御目标是建立稳定的核验流程，让学习者能从固定案例中识别风险线索，并选择举报、隔离、二次确认或放行。

本实验当前处于 `in-progress`，已接入后端固定案例 `review` API。以下内容同时作为当前 API 约束和后续前端实现约束。

## 2. 固定检查清单

当前修复版固定案例 API 已覆盖：

- 发件人显示名与实际域名一致性。
- 链接展示文本与目标域名一致性。
- 凭据、验证码、付款、下载或远程协助诱导。
- 附件类型与业务上下文。
- 请求是否绕开正常流程。
- 是否需要使用安全通道二次确认。

## 3. 后端判定

当前 API 只允许读取固定字段：

- `caseKey`
- `reviewModeKey`
- `defenseChecklistKey`

判定结果只返回固定学习信号、建议动作和风险摘要，不返回完整邮件模板或外部目标。

当前接口：

```text
POST /api/labs/social/phishing/:variant/review
```

漏洞版固定风险案例会展示错误放行倾向；修复版固定风险案例会返回阻断 / 举报 / 隔离建议；固定安全案例会返回 `phishing-safe-message-accepted`。

## 4. 日志摘要建议

当前统一事件日志只记录：

- 固定案例 key。
- 检查模式 key。
- 检查清单 key。
- 命中的风险标签。
- 建议动作。
- 学习信号。

不得记录完整邮件正文、真实邮箱、真实链接、真实附件、凭据、验证码、Cookie 或 token。

## 5. 后续 ready 边界

如果后续标记为 `ready`，必须满足 case-study ready 例外标准：

- 保持 `mode: "case-study"`。
- 漏洞版与修复版都有可访问学习入口。
- 自动化验证至少覆盖两类证据。
- 文档明确禁止真实服务连接、真实凭据、对外攻击脚本、payload 库和任意执行器。
- `variants[].supportsAutomation` 不得因页面或只读验证存在而误标为攻击脚本自动化。
