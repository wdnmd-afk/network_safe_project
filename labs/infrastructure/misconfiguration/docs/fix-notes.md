# 配置错误修复说明

## 1. 修复目标

配置错误的修复重点不是把所有服务全部关闭，而是区分业务入口、管理入口、调试能力、错误信息和跨域信任边界，让必要能力在受控范围内可用。

当前为 `in-progress` 阶段，后端受控 `audit` API 已能返回固定修复审计结果。本文件只记录修复思路和 API 阶段观察点，不提供真实配置修改命令、重载命令、部署步骤或系统操作。

## 2. 建议修复策略

- 默认关闭调试入口、目录索引和公开状态页。
- 管理入口必须要求认证、授权和来源限制。
- CORS 使用明确可信来源、方法、头部和凭据策略，不使用宽泛放行。
- 用户可见错误信息只保留必要说明，详细堆栈进入受控日志。
- 默认凭据提示必须移除，初始化流程应强制设置安全凭据。
- 配置变更进入审计清单，发布前检查暴露面和安全边界。

## 3. 当前修复版观察点

当前修复版 API 可展示：

- 暴露面从高风险状态收敛为最小必要状态。
- 管理入口从公开可见变为认证与来源限制。
- CORS 从过宽策略变为最小权限策略。
- 详细错误信息从用户页面移入内部日志摘要。
- 正常业务入口仍可保留，不把防御理解成“全部阻断”。

## 4. 日志与数据边界

后续事件日志只允许记录：

- `configCaseKey`
- `auditPolicyKey`
- `exposureCategory`
- `riskIndicatorCount`
- `auditActions`
- `matchedControlledSample`
- `signal`

不得记录：

- 完整配置文件。
- 真实路径、真实服务地址或真实端口清单。
- 真实账号、密码、token、Cookie、证书或数据库连接串。
- 本机环境、云账号或服务指纹。

## 5. 真实生产补充

真实生产环境还需要结合：

- 配置基线和变更审批。
- 环境隔离与最小权限。
- 运行时安全监控。
- 凭据轮换与密钥管理。
- 资产清单和暴露面管理。

这些内容在本项目中只做学习说明，不连接真实基础设施。
