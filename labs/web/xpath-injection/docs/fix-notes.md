# XPath 注入修复说明规划

## 1. 漏洞根因

XPath 注入的核心风险来自把用户输入拼接进查询条件结构。

当输入不再只是文本值，而是影响查询路径、过滤条件或范围时，攻击方可能让结果范围被扩大。

## 2. 修复策略

本项目后续修复版采用受控模拟方式展示以下策略：

- 查询模板固定在服务端。
- 可查询字段和查询范围使用允许列表。
- 用户输入只作为文本值处理。
- 受控风险样例在进入虚拟查询前被阻断。
- 日志只保存风险摘要和学习信号。

## 3. 不采用的方式

本实验不采用以下方式作为修复重点：

- 只依赖前端校验。
- 让客户端提交 XPath 表达式。
- 在后端执行任意 XPath 表达式后再过滤结果。
- 读取真实 XML 文件进行演示。
- 使用外部 XML 数据源。

## 4. 日志要求

后续事件日志建议记录：

- `queryTemplate`
- `scope`
- `keywordLength`
- `keywordPreview`
- `detectedRiskTypes`
- `matchedControlledSample`
- `resultScope`
- `documentCount`
- `signal`

不得记录：

- 完整关键词。
- 完整 XPath 表达式。
- 完整危险样例。
- 真实 token、Cookie 或外部目标信息。
- 前端可见的原始输入摘要 JSON。

## 5. 生产环境补充

真实生产环境还需要结合具体 XML 处理库补充：

- 安全的 XPath 参数绑定或等价安全 API。
- XML 解析器安全配置。
- 最小权限数据读取。
- 审计日志与异常告警。
- 对业务字段和返回范围的服务端约束。

这些内容会在本实验中以说明方式呈现，不直接引入真实 XML 查询执行器。
