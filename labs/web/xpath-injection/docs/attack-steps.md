# XPath 注入攻击步骤规划

## 1. 攻击方视角

攻击方关注的是：业务关键词是否会影响 XML 查询条件的结构，而不仅仅作为普通文本值参与匹配。

本实验后续只使用固定受控学习样例：

```text
LAB_CONTROLLED_XPATH:expand-product-catalog
```

该样例是教学标识，不是通用 XPath 表达式。

## 2. 计划步骤

1. 进入漏洞版页面。
2. 使用正常关键词查询公开产品目录。
3. 观察后端返回 `xpath-injection-safe-query-completed`。
4. 填入固定受控学习样例。
5. 提交查询并观察漏洞版返回 `xpath-injection-result-scope-expanded`。
6. 对比虚拟结果中公开结果和教学模拟的内部结果。
7. 打开账户中心或实验详情页复盘事件日志摘要。
8. 切换到修复版页面，提交同样样例。
9. 观察修复版返回 `xpath-injection-controlled-sample-blocked`。

## 3. 观察点

- 漏洞版正常查询是否只返回公开匹配结果。
- 漏洞版受控样例是否扩大虚拟结果范围。
- 修复版是否在服务端阻断受控样例。
- 事件日志是否只展示摘要字段。
- 页面是否没有展示完整危险输入或原始输入摘要 JSON。

## 4. 边界提醒

- 不对外部站点使用本实验样例。
- 不把教学标识改写成真实 XPath 表达式。
- 不读取真实 XML 文件。
- 不连接外部 XML 数据源。
- 不保存真实凭据、token、Cookie 或外部目标信息。
