# XSS 修复说明

## 修复目标

让用户提交的客服留言只作为文本内容展示，不能改变页面结构。

## 当前修复策略

修复版页面使用文本渲染展示留言内容。同样的样例 payload：

```html
<mark data-xss-lab-signal="xss">XSS 模拟信号</mark>
```

在修复版中会原样显示为字符串，不会生成新的 HTML 标记。

## 为什么这样修复

XSS 的核心风险来自“不可信输入进入可执行或可解析上下文”。在当前客服留言场景中，业务并不需要富文本能力，因此最小、清晰、可验证的修复方式是文本渲染。

## 常见误区

- 只过滤 `<script>` 不够，HTML 事件属性、URL 协议和 SVG 等上下文也可能产生风险。
- 只拦截一个固定 payload 不够，修复应针对输出上下文，而不是针对样例字符串。
- 在可信和不可信内容混合时直接使用 HTML 渲染，会让后续维护者难以判断安全边界。

## 后续富文本场景

如果后续业务确实需要富文本，应单独设计：

- 允许的标签白名单。
- 允许的属性白名单。
- URL 协议过滤。
- 自动化测试覆盖危险标签、事件属性和协议。
- 清晰标注输入来源与输出上下文。

当前阶段不引入富文本清洗依赖，避免扩大样板实验范围。
