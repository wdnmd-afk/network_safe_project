# 文件上传修复说明

## 当前修复策略

修复版在入口层同时检查三个条件：

- 扩展名白名单：只允许 `.png`、`.jpg`、`.jpeg`、`.webp`。
- MIME 白名单：只允许 `image/png`、`image/jpeg`、`image/webp`。
- 内容特征：拦截 `<?php`、`<script`、`<%`、`system(`、`exec(` 等可执行内容特征。

只要任一条件不可信，接口就返回 403，并记录 `file-upload-validation-blocked`。

## 为什么不能只校验一个字段

- 只看扩展名：攻击者可能使用双扩展名、大小写绕过或服务端解析差异。
- 只看 MIME：客户端可伪造 `Content-Type`。
- 只看内容特征：规则不可能覆盖所有语言、模板和解释器行为。

教学版采用多条件并行校验，是为了让学习者先建立“上传入口必须默认不可信”的判断习惯。

## 后续真实上传还需要补充

- 限制文件大小和请求体大小。
- 使用随机文件名，避免覆盖和路径猜测。
- 将上传内容存到不可执行目录或独立对象存储。
- 图片类上传应解码后重写，丢弃原始危险结构。
- 下载域名与业务域名隔离，降低脚本执行风险。
- 记录审计日志，但避免保存完整敏感内容。
- 高风险上传可进入人工审核或异步扫描。

## 当前实验刻意不做的事

- 不保存真实上传文件。
- 不生成可执行 WebShell。
- 不提供绕过修复策略的通用技巧集合。
- 不面向任何外部真实目标。
