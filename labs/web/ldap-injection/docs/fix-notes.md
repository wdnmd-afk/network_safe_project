# LDAP 注入修复说明

## 修复原则

LDAP 注入防护的核心原则是：用户输入不能改变目录查询结构。

修复不是靠前端隐藏输入框，也不是只靠提示用户不要输入特殊字符。所有关键边界必须由服务端控制。

## 1. 固定查询模板

服务端应根据业务场景选择固定模板：

- 组织成员搜索模板。
- 权限组查询模板。
- 登录候选用户定位模板。

模板决定查询结构，用户输入只能进入预留的文本值位置。

## 2. 字段允许列表

允许查询的字段必须固定：

- 成员搜索只允许固定展示字段。
- 权限组查询只允许固定组名字段。
- 登录筛选只允许固定身份标识字段。

客户端不能提交目录字段名、权限范围条件或查询结构。

## 3. 过滤器转义

进入目录过滤条件的文本值必须进行上下文转义，使输入保持普通文本语义。

本项目当前只提供受控虚拟目录 API，不实现真实转义函数，也不连接真实 LDAP 服务。虚拟 API 用固定场景和受控样例说明防护差异，不能替代生产级过滤器转义实现。

## 4. 权限边界

目录查询必须叠加服务端权限判断：

- 当前组织范围来自登录态或服务端上下文。
- 权限组可见范围来自服务端角色判断。
- 登录流程不能因为候选用户定位异常而绕过凭据校验。

## 5. 最小权限

真实生产中的目录查询账号应只拥有必要读取权限。即使查询条件出现异常，也不应能读取超出业务需要的数据。

## 6. 审计与告警

应记录脱敏审计信息：

- 查询场景。
- 当前用户或服务账号的脱敏标识。
- 风险标签。
- 结果范围是否异常。
- 阻断或放行决策。

当前前端工作台调用的虚拟 API 已通过统一事件日志记录这些安全摘要，并且不保存原始关键词或过滤器字符串。

不得记录完整危险输入、LDAP 过滤器字符串、目录账号、DN、邮箱、手机号、凭据或外部目标信息。

## 7. 登录场景补充

登录流程还需要：

- 独立凭据校验。
- 失败计数。
- 账号锁定或节流。
- 异常候选用户数量告警。
- 服务端统一日志。
