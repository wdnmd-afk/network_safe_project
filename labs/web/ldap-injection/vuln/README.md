# LDAP 注入漏洞案例

## 目标

漏洞案例用于说明：当用户输入被拼进目录服务过滤条件时，查询语义可能偏离原本业务意图。

本目录不包含真实目录服务连接，也不提供可复制的过滤器 payload。当前后端只提供受控虚拟目录 API，用固定场景展示结果范围被扩大的学习信号。

前端漏洞版工作台可提交固定 `scenarioKey` 和 `keyword`。正常关键词用于观察公开范围查询，页面提供的受控学习样例用于观察 `ldap-injection-scope-expanded`，该样例不是通用 LDAP 过滤器。

## 案例一：组织成员搜索

业务意图：

- 用户只能在当前组织范围内搜索成员。
- 姓名关键词只应作为普通文本值参与匹配。

漏洞观察点：

- 攻击方会先判断姓名关键词是否影响过滤条件结构。
- 如果组织范围不是服务端固定条件，输入可能让查询范围偏离当前组织。
- 风险不是“看到更多文字”本身，而是身份边界和组织边界被输入影响。

## 案例二：权限组查询

业务意图：

- 普通用户只能查看被允许的权限组。
- 组名关键词不应改变权限边界。

漏洞观察点：

- 攻击方会关注组名输入是否能影响权限过滤逻辑。
- 如果服务端接受客户端传入过滤结构，权限组可见范围可能被扩大。
- 日志应记录风险标签和案例 key，而不是保存完整危险输入。

## 案例三：登录筛选

业务意图：

- 登录流程按用户名定位候选用户。
- 候选用户定位后仍必须进行独立凭据校验。

漏洞观察点：

- 攻击方会关注用户名是否参与可拼接过滤条件。
- 过滤条件语义偏移可能影响候选用户定位。
- 登录安全还依赖失败计数、审计、账号策略和最小权限。

## 禁止行为

- 不展示真实 LDAP 过滤器字符串。
- 不连接真实目录服务。
- 不保存目录账号、DN、邮箱、手机号或凭据。
- 不提供对外查询脚本。
- 不生成 payload 库。
- 不把受控学习样例作为真实 LDAP 过滤器使用。
