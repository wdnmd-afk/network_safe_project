# 技术选型与版本策略

## 1. 文档目标

本文档用于明确本项目在进入工程实现前的技术基线与版本策略，重点解决以下问题：

1. 前端、后端、数据库、测试、脚本分别选什么
2. 为什么这样选，而不是简单追求“全部最新版”
3. 哪些技术应优先跟随稳定主线，哪些应锁定 LTS
4. 后续如何升级，避免破坏漏洞版 / 修复版实验与自动化验证

本文档属于规划阶段基线文档。后续实际初始化工程时，必须以本文档为准落地版本声明、锁文件和升级规则。

## 2. 选型原则

### 2.1 总体原则

- 优先选择官方仍在支持范围内的稳定版本
- 优先选择生态成熟、资料充足、便于本地 Windows 环境运行的技术
- 优先选择能清晰区分“平台核心代码”和“漏洞实验代码”的技术组合
- 优先选择便于后续自动化测试接入的技术

### 2.2 版本原则

- **运行时与数据库**：优先使用 LTS 或稳定生产线
- **前端框架与构建工具**：优先使用当前稳定主线
- **实验场景相关依赖**：优先保证可控、可复现、便于演示漏洞版 / 修复版差异
- **预览版、RC、实验特性**：一期默认不使用

### 2.3 本项目的现实约束

- 运行环境是 Windows 本机
- 暂不使用 Docker
- MySQL 跑在本机 `localhost`
- 前端允许 `build` 后交由 `nginx` 托管
- 后端 Node 服务独立运行
- 仓库采用 monorepo

因此，本项目不适合使用过重的本地编排方案，也不适合采用对 Linux 容器环境依赖很强的默认设计。

## 3. 总体技术基线

## 3.1 monorepo 与包管理

### 选型

- 包管理器：`pnpm`
- monorepo 组织：`pnpm workspace`

### 结论

- 一期先不引入 `Nx`、`Turborepo` 这类额外编排层
- 后续只有在构建、缓存、任务编排复杂度明显上升时，再评估是否补充

### 原因

- 当前项目仍处于规划与骨架阶段，先保持结构简单
- `pnpm workspace` 足够支持：
  - `apps/web`
  - `apps/server`
  - `packages/shared`
  - `packages/configs`
  - `packages/testing`
- 对 Windows 本机开发者更直接，排障成本更低

## 3.2 前端技术栈

### 选型

- 框架：`Vue 3`
- 构建工具：`Vite`
- 路由：`Vue Router`
- 状态管理：`Pinia`
- 语言：`TypeScript`
- HTTP 客户端：后续优先使用 `fetch` 封装，必要时再评估 `axios`

### 建议版本基线

- `vue`: `3.5.x`
- `vite`: `8.x`
- `vue-router`: `5.x`
- `pinia`: `3.x`
- `typescript`: `6.0.x`

### 原因

- Vue 3 当前仍是主流稳定路线，适合长期维护
- Vite 在 Vue 生态中的默认地位已经非常稳定，适合本地开发和 nginx 静态部署
- Vue Router 5 已经进入当前稳定主线，适合以新项目基线直接采用
- Pinia 是 Vue 官方推荐状态管理方案，适合实验列表、学习记录、验证记录等前端状态建模
- TypeScript 作为全栈统一类型基础，有利于共享实验元数据、API DTO 和测试辅助类型

### 前端边界原则

- 不在一期引入过重的 UI 框架
- 优先自己建立轻量平台 UI 层，避免依赖大型后台模版体系
- 允许后续按需补充图表、代码高亮、Markdown 渲染等学习型依赖，但必须按实验平台需要引入

## 3.3 后端技术栈

### 选型

- 运行时：`Node.js`
- Web 框架：`Express`
- 语言：`TypeScript`
- 数据访问：
  - 平台核心能力优先使用 `Prisma`
  - 需要明确演示注入风险或原生 SQL 差异的实验场景允许使用 `mysql2`

### 建议版本基线

- `node`: `22.x LTS`
- `express`: `5.1.x`
- `typescript`: `6.0.x`
- `prisma`: `6.x`
- `@prisma/client`: `6.x`
- `mysql2`: `3.x`

### 原因

- Node 22 仍属于官方 LTS 线，适合作为稳定运行时基线
- 你已明确要求本项目使用 Node 22，因此文档基线按该要求统一
- Node 26 虽然已发布，但当前仍是 Current 线，不应作为一期默认生产基线
- Express 5 已正式稳定，学习成本低，便于构建清晰的实验路由与平台 API
- Prisma 适合平台核心表、迁移和类型安全
- `mysql2` 适合少量保留原生 SQL 的场景，便于真实展示 SQL 注入前后差异

### 后端边界原则

- 平台核心功能不应为了“做漏洞”而故意写成整体不安全结构
- 漏洞实验代码应明确隔离在实验模块或实验服务层
- 能通过参数化查询、白名单、服务封装解决的平台逻辑，默认走安全实现
- 只有实验场景中的漏洞版代码才允许保留受控不安全实现

## 3.4 数据库技术栈

### 选型

- 数据库：`MySQL`
- 本机连接方式：`localhost`
- ORM：`Prisma`
- 原生驱动：`mysql2`

### 建议版本基线

- `MySQL 8.4 LTS`

### 原因

- MySQL 官方已区分 Innovation 与 LTS 发布轨道
- 本项目优先追求稳定、可复现、便于本机维护
- 8.4 LTS 比 Innovation 线更适合作为长期学习平台基线
- 后续实验需要长期保留、反复验证，数据库主线不宜频繁波动

### 数据库策略

- 平台核心表结构应优先稳定，不随单个实验频繁变动
- 实验专用数据表允许按实验模块拆分
- 漏洞版与修复版尽量共用主题数据模型，但通过不同查询方式、校验方式、权限方式区分
- 后续数据库文档中应明确哪些表属于平台核心，哪些表属于实验场景

## 3.5 自动化测试技术栈

### 选型

- 单元 / 模块测试：`Vitest`
- API / 集成测试：`Vitest` + Node 测试辅助
- E2E 测试：`Playwright`
- 实验脚本验证：`Python` / `TypeScript`

### 建议版本基线

- `vitest`: `4.1.x`
- `playwright`: `1.59.x`

### 原因

- Vitest 与 Vite / TypeScript 集成自然，适合前后端共享测试工具链
- Playwright 适合：
  - 漏洞版与修复版页面行为对照
  - 登录、授权、记录、学习流程验证
  - 前端 build 后在 nginx 下的回归验证
- Python / TypeScript 脚本更适合承载实验级验证、样本生成和辅助攻击流程演示

### 测试分层原则

- 平台层：测试正常业务与安全控制逻辑
- 实验层：测试漏洞是否可触发、修复是否生效
- 脚本层：测试实验辅助脚本是否在受控场景中可运行

## 3.6 实验脚本技术栈

### 选型

- `Python 3.12+`
- `TypeScript`

### 原因

- Python 适合快速编写本地实验脚本、数据生成脚本、受控验证脚本
- TypeScript 适合与平台共享类型、与 Node 测试体系衔接
- 两者组合能覆盖：
  - 受控验证
  - 样本构造
  - 场景驱动脚本
  - 自动化测试辅助逻辑

### 目录原则

脚本统一放在：

```text
tools/lab-scripts/<category>/<scene>/
```

不把脚本散落到前端或后端业务目录中。

## 3.7 反向代理与部署

### 选型

- 前端静态托管：`nginx`
- 后端服务：`Node.js` 独立运行

### 原因

- 符合当前已确认约束
- 便于分别验证：
  - 开发环境下的实验效果
  - `build` 后静态站点下的实验效果
- 后续可以明确区分：
  - 前端静态资源安全策略
  - 后端接口安全策略
  - nginx 层安全头与转发配置

## 4. 为什么不是“全部追最新”

本项目虽然要求技术尽量新，但“新”必须服从以下目标：

1. 本机 Windows 环境可稳定运行
2. 漏洞版 / 修复版实验可持续复现
3. 自动化测试结果可重复
4. 文档、代码、脚本三者长期一致

因此，本项目采用以下策略：

- **运行时与数据库**：优先 LTS
- **前端框架与构建工具**：优先当前稳定主线
- **测试工具**：优先当前稳定主线，但避免预览版
- **实验依赖**：优先可控和易隔离，而不是盲目追新

这比“所有东西都装最新版本”更适合安全学习平台。

## 5. 版本锁定策略

## 5.1 强锁内容

以下内容必须明确锁定：

- Node 主版本
- MySQL 主版本
- `pnpm` 版本
- `packageManager` 字段
- `engines.node` 字段
- 锁文件 `pnpm-lock.yaml`

### 目的

- 保证本机复现一致
- 保证前后端依赖解析一致
- 保证自动化测试结果稳定

## 5.2 中锁内容

以下内容采用“稳定 major + 受控 minor”策略：

- Vue
- Vite
- Vue Router
- Pinia
- Express
- Prisma
- Playwright
- Vitest

### 目的

- 保持技术不过旧
- 也避免每次小升级都破坏实验行为

## 5.3 谨慎升级内容

以下内容升级时必须单独验证：

- 认证相关库
- 文件上传相关库
- 模板引擎
- XML 解析库
- Markdown 渲染库
- 与实验攻击面直接相关的第三方依赖

### 原因

- 这些依赖一旦升级，可能直接改变漏洞是否可触发、修复是否仍然成立、测试断言是否失效

## 6. 升级策略

## 6.1 升级节奏

- Node / MySQL：按季度评估，不频繁升级
- 前端与测试工具：按小版本节奏评估
- 安全敏感依赖：按场景逐项评估

## 6.2 升级前必须确认

1. 是否影响现有漏洞版行为
2. 是否影响修复版验证逻辑
3. 是否影响自动化测试脚本
4. 是否影响 Windows 本机运行方式
5. 是否影响 nginx 静态部署效果

## 6.3 升级验证最低要求

每次关键升级后，至少验证以下内容：

1. 平台首页和实验列表可正常打开
2. 至少一个 Web 漏洞场景的漏洞版可复现
3. 同一场景修复版防护生效
4. 登录、学习记录、验证记录链路可正常工作
5. 至少一条 Playwright 测试通过
6. 至少一条实验脚本验证通过

## 9. 当前结论

本项目的技术选型策略不是“保守到过时”，也不是“激进到不稳定”，而是：

- 用较新的稳定主线建设前端与测试工具
- 用 LTS 运行时和 LTS 数据库保证长期可复现
- 用 ORM + 原生驱动分层隔离平台代码与漏洞实验代码
- 用统一脚本目录与测试分层为后续自动化测试铺路

后续进入工程初始化时，必须先按本文档建立版本声明与锁定策略，再开始代码骨架落地。
