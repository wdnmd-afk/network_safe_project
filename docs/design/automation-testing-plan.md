# 自动化测试规划

## 1. 文档目标

本文档用于定义本项目的一期自动化测试策略，解决以下问题：

1. 自动化测试需要覆盖哪些层
2. `Vitest`、`Playwright`、`Python/TypeScript` 脚本分别负责什么
3. 漏洞版验证与修复版回归验证如何区分
4. 一期最小可落地的自动化范围是什么

## 2. 测试设计原则

### 2.1 测试目标不是“追求数量”，而是验证平台模式成立

一期自动化测试的核心目标是证明以下能力已经成立：

1. 平台可以稳定注册和展示实验
2. 漏洞版与修复版能被明确区分
3. 至少一个实验可以被自动化验证
4. 学习记录与验证记录的基础链路可自动验证

### 2.2 测试分层必须清晰

本项目自动化测试分为三层：

- **单元 / 模块层**
- **API / 集成层**
- **端到端 / 场景层**

另有一层辅助能力：

- **实验脚本验证层**

### 2.3 漏洞版与修复版必须分别验证

自动化测试不能只验证“页面能打开”。

对于支持漏洞版 / 修复版对照的实验，至少要验证：

1. 漏洞版存在预期不安全行为或可观察信号
2. 修复版不存在该不安全行为，或出现拦截 / 编码 / 权限拒绝等预期防护信号

### 2.4 测试要服务于本机 Windows 场景

一期测试设计必须默认适配：

- Windows 本机运行
- MySQL 本机 `localhost`
- 前端开发环境
- 后端本地 Node 运行
- build 后 nginx 托管验证

## 3. 测试工具分工

## 3.1 `Vitest`

### 职责

- 共享类型与元数据校验测试
- 前端工具函数与状态逻辑测试
- 后端服务层与路由处理逻辑测试
- 实验注册与索引逻辑测试
- API 层最小集成测试

### 一期优先覆盖

- `meta.json` 解析与校验
- 实验扫描与索引生成
- 平台实验列表服务
- 平台实验详情服务
- 学习记录与验证记录服务层

### 不负责的内容

- 浏览器真实交互行为
- 漏洞页面的最终用户可见效果

## 3.2 `Playwright`

### 职责

- 前端页面与后端接口联动验证
- 漏洞版 / 修复版 UI 对照验证
- 学习流程、记录流程、验证流程的端到端验证
- build 后前端在 nginx 下的回归验证

### 一期优先覆盖

- 平台首页
- 实验列表页
- 实验详情页
- 第一个样板实验：`web/xss`

### 不负责的内容

- 大量底层纯函数测试
- 脚本级实验资源验证

## 3.3 `Python / TypeScript` 脚本

### 职责

- 受控实验脚本验证
- 样本生成
- 初始化场景数据
- 对不适合纯页面操作的场景进行辅助验证

### 一期优先覆盖

- 实验种子数据初始化
- XSS、命令注入、SSRF 等可能需要的辅助请求脚本
- 后续网络模拟实验的本机脚本验证

### 不负责的内容

- 平台 UI 回归
- 平台级通用状态逻辑

## 4. 测试目录规划

## 4.1 前端 / 共享 / 后端测试

建议目录：

```text
apps/web/tests/
apps/server/tests/
packages/shared/tests/
packages/testing/
```

## 4.2 E2E 测试

建议目录：

```text
apps/web/tests/e2e/
apps/web/tests/labs/
```

说明：

- `e2e/`：平台级流程
- `labs/`：实验级端到端验证

## 4.3 脚本验证

建议目录：

```text
tools/lab-scripts/<category>/<scene>/
```

推荐内容：

- `README.md`
- `exploit.py`
- `verify.ts`
- `seed.ts`
- `samples/`
- `artifacts/`

## 5. 一期测试分层规划

## 5.1 单元 / 模块层

### 目标

验证平台基础能力和实验注册机制。

### 首批测试主题

1. 元数据字段合法性校验
2. 实验目录扫描结果正确性
3. 实验索引构造逻辑
4. 学习记录写入逻辑
5. 验证记录写入逻辑

### 工具

- `Vitest`

## 5.2 API / 集成层

### 目标

验证平台后端最小 API 链路。

### 首批测试主题

1. 健康检查接口
2. 实验列表接口
3. 实验详情接口
4. 学习记录接口
5. 验证记录接口

### 工具

- `Vitest`
- 后续按需配合测试数据库

## 5.3 端到端层

### 目标

验证用户真实可见流程。

### 首批测试主题

1. 首页可打开
2. 实验列表可加载
3. 实验详情可打开
4. `web/xss` 漏洞版与修复版可区分

### 工具

- `Playwright`

## 5.4 脚本验证层

### 目标

验证实验脚本在本地受控环境中可运行。

### 首批测试主题

1. XSS 场景种子脚本
2. XSS 场景验证脚本
3. 后续命令注入 / SSRF / SQL 注入辅助脚本

### 工具

- `Python`
- `TypeScript`

## 6. 漏洞版与修复版如何测试

## 6.1 漏洞版测试目标

漏洞版测试不等于“攻击外部目标”，而是验证本地受控实验中的可观察不安全行为。

示例：

- XSS：恶意输入被执行或产生预期 DOM 变化
- SQL 注入：恶意条件导致返回异常扩大的结果集
- IDOR：越权访问成功

## 6.2 修复版测试目标

修复版测试要验证防护已经生效。

示例：

- XSS：恶意输入被编码为纯文本
- SQL 注入：恶意输入不再扩大结果集
- IDOR：越权访问被拒绝

## 6.3 对照断言原则

对于支持双版本的实验，自动化测试优先采用对照式断言：

1. 同样的输入访问漏洞版
2. 观察漏洞版产生预期不安全信号
3. 同样的输入访问修复版
4. 观察修复版不再出现该信号

这样比只测“修复版没报错”更有价值。

## 7. 一期首批自动化覆盖清单

一期不要求一次写完所有测试，先覆盖最关键的一条主链路。

## 7.1 平台基础

- [ ] 实验元数据校验测试
- [ ] 实验列表服务测试
- [ ] 实验详情服务测试
- [ ] 学习记录服务测试
- [ ] 验证记录服务测试

## 7.2 样板实验

- [ ] `web/xss` 漏洞版页面验证
- [ ] `web/xss` 修复版页面验证
- [ ] `web/xss` 对照断言测试
- [ ] `web/xss` 种子脚本验证

## 7.3 build 后回归

- [ ] 前端 build 后 nginx 场景下首页可访问
- [ ] 前端 build 后样板实验入口可访问

## 8. 与元数据的联动

测试设计必须与 `meta.json` 对齐。

后续测试应尽量从元数据中读取：

- `id`
- `slug`
- `variants`
- `entrypoints`
- `verification`
- `paths`

### 原因

- 减少硬编码路径
- 降低目录调整成本
- 保证测试和实验注册使用同一份定义

## 9. 测试环境约定

一期建议至少区分三种运行上下文：

1. **开发环境**
   - 前端 dev server
   - 后端本地 server
   - MySQL localhost
2. **本机集成环境**
   - 前后端联调
   - 平台记录与实验数据联通
3. **本机部署验证环境**
   - 前端 build 后由 nginx 托管
   - 后端继续本地运行

## 10. 一期最小验证标准

当以下条件成立时，可以认为一期自动化测试已经具备最小可用价值：

1. 至少一个实验能被平台注册并展示
2. 至少一个实验具备 `Vitest` 校验
3. 至少一个实验具备 `Playwright` 对照验证
4. 至少一个实验具备脚本验证入口
5. 平台能记录一条学习记录
6. 平台能记录一条验证记录

## 11. 不建议的做法

- 不建议先写大量 Playwright 用例再补平台结构
- 不建议把所有实验都先做成脚本验证而不接平台
- 不建议只写漏洞版自动化，不写修复版回归
- 不建议测试文件完全脱离 `meta.json`
- 不建议一期就追求全量覆盖全部实验

## 12. 后续落地顺序

自动化测试落实建议按以下顺序推进：

1. 先做元数据与实验注册相关 `Vitest`
2. 再做平台最小 API / 服务层测试
3. 再做 `web/xss` 的 `Playwright` 对照验证
4. 再补 `web/xss` 的脚本验证
5. 最后把模式复制到其他高优先级实验

## 13. 当前结论

一期自动化测试不追求“大而全”，而要先证明平台模式已经成立：

- 平台能识别实验
- 平台能展示实验
- 平台能区分漏洞版与修复版
- 平台能记录学习与验证
- 至少一个实验能被三层测试中的多层方式覆盖
