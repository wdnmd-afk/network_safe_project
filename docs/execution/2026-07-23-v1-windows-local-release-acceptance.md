## 1. 目标

验证当前 65 个安全学习实验平台是否能够在 Windows 本机环境中，以可重复的方式完成：

1. 数据库创建、迁移、复盘表补齐和种子初始化。
2. 前端生产构建和静态资源产出。
3. 后端生产构建、启动和数据库探活。
4. nginx 静态托管、SPA history fallback 和 `/api` 反向代理。
5. 登录、实验目录、实验详情、漏洞版/修复版和事件日志的代表性验收。
6. Smoke、代表性 Playwright 和发布后清理流程。

本验收只针对本机、固定数据和受控学习场景，不访问外部目标，不读取真实凭据，不执行真实攻击行为。

## 2. 范围

### 2.1 纳入范围

- `apps/web` 前端生产构建和预览。
- `apps/server` 后端生产构建和启动。
- `database/migrations` 全量迁移。
- `apps/server/scripts/seed-auth-users.mjs` 认证种子。
- `apps/server/scripts/seed-lab-metadata.ts` 实验种子。
- `apps/server/scripts/ensure-local-schema.mjs` 缺失表幂等补齐。
- nginx 配置、静态目录和 API 代理。
- `packages/testing` Smoke 与代表性 E2E。
- 健康检查、实验目录、登录和至少一个漏洞版/修复版差异。
- 验收证据、失败排查和清理步骤。

### 2.2 不纳入范围

- 公网部署、域名、HTTPS 证书和外部服务。
- Docker、云服务或远程数据库。
- 真实邮件、短信、支付、浏览器凭据或第三方平台。
- 全量浏览器逐页人工验收；逐项场景证据继续由已有脚本和服务测试承担。

## 3. 前置条件

- Node.js 满足根 `package.json` 的 `>=22 <23`。
- pnpm 版本与 `packageManager` 一致。
- 本机 MySQL 服务可用，数据库连接信息只从 `apps/server/.env` 或示例配置生成。
- nginx 已安装，或通过明确的本机路径提供可执行文件。
- 端口约定保持：MySQL `3306`、后端 `6667`、开发前端 `6670`、nginx `8080`。
- `LT-002` 数据库初始化流程已经完成。
- `LT-003` nginx 配置与部署说明已经完成。

## 4. 实施步骤

### 4.1 预检

1. 检查 Node、pnpm、MySQL、nginx 版本和可执行路径。
2. 检查 `apps/server/.env.example` 和本机 `.env` 必需变量。
3. 检查目标端口是否被占用。
4. 检查工作区是否存在未预期的用户修改。
5. 检查数据库迁移目录、种子入口和 nginx 配置路径。

### 4.2 数据库准备

1. 创建本机数据库（不存在时幂等创建）。
2. 按文件名顺序应用 `database/migrations/*.sql`。
3. 执行 `schema:ensure`，确认复盘题完成表存在。
4. 执行 `seed:auth`，确认演示账号可用。
5. 执行 `seed:labs`，确认 9 个分类、65 个实验和 130 个变体同步。
6. 执行数据库健康检查。

### 4.3 构建

1. 执行前端生产构建，确认 `apps/web/dist` 生成入口和静态资源。
2. 执行后端生产构建，确认 Prisma client 和 `apps/server/dist` 生成入口。
3. 检查构建产物未包含 `.env`、凭据、浏览器数据、缓存或无关生成文件。
4. 记录构建命令、Node/pnpm 版本和产物摘要。

### 4.4 启动

1. 使用构建后的 Node 入口启动后端。
2. 检查 `/api/health` 和 `/api/health/db`。
3. 使用 nginx 托管前端 `dist`。
4. 检查 `/`、`/labs` 和任意深层实验路径刷新不会返回 404。
5. 检查 nginx `/api/*` 请求正确代理到 `6667`。

### 4.5 功能验收

至少验证：

- [x] 演示用户登录、当前用户和退出。
- [x] 实验分类列表和实验详情。
- [x] 一个专用实验漏洞版与修复版差异。
- [x] 一个通用引导式实验漏洞版与修复版差异。
- [x] 事件日志写入和安全摘要展示。
- [x] 账户中心复盘和问题完成状态。
- [x] 平台状态页和数据库探活。

### 4.6 自动化验收

1. 运行服务端、前端、共享包和测试基础设施测试。
2. 运行 `test:guided`，逐项检查 38 个引导式场景。
3. 运行 Smoke，确认构建后服务和 nginx 代理链路。
4. 运行代表性 Playwright，覆盖专用实验、通用实验和跨分类入口。
5. 运行元数据、路由、API、脚本和文档一致性检查。
6. 运行 `git diff --check` 和危险能力扫描。

### 4.7 清理和复现

1. 停止 Node 和 nginx 进程。
2. 确认测试临时进程、端口和日志已清理。
3. 保留不含秘密的验收摘要。
4. 在同一台机器上重新执行初始化和启动关键步骤，确认幂等。

## 5. 实施建议

- 数据库迁移脚本必须具备顺序记录，避免重复执行导致状态不明确。
- nginx 配置使用仓库相对路径说明和可替换的本机绝对路径，不把个人路径写入代码。
- 构建和验收命令优先封装为根级脚本，减少手工步骤差异。
- 生产构建只在本任务授权范围内执行，日常开发继续使用局部验证。
- 验收账号只使用固定演示账号，密码不得写入提交记录、日志或截图。
- 代表性 E2E 只点击固定控件，不提供任意输入或外部目标。

## 6. 潜在风险分析

| 风险 | 影响 | 控制方式 |
|---|---|---|
| MySQL 未运行或权限不足 | 初始化无法开始 | 预检端口、明确报错并提供本机排查步骤 |
| 迁移中途失败 | 数据库处于半初始化状态 | 记录迁移表，失败时停止并保留可诊断日志 |
| nginx 路径依赖个人机器 | 新机器无法复现 | 使用示例配置和变量化路径，不提交个人路径 |
| 构建产物泄露秘密 | 造成配置风险 | 构建后扫描 `.env`、token、Cookie 和缓存 |
| 旧进程占用端口 | 验收结果不可信 | 预检并记录进程，禁止静默复用未知服务 |
| 仅开发服务器通过 | 不能证明发布链路 | 必须验证构建产物、Node `start` 和 nginx |
| E2E 写入真实数据 | 污染个人学习记录 | 使用固定演示账号并在验收前后清理或隔离数据 |

## 7. 优化方案

- 增加 `release:check` 组合入口，统一预检、元数据审计、类型检查、测试和 Smoke。
- 增加 `db:prepare` 入口，串联迁移、schema ensure、认证种子和实验种子。
- 增加 nginx 示例配置和 Windows PowerShell 启停说明。
- 让 Smoke 能识别“构建后服务”和“开发服务”两种运行上下文。
- 验收摘要只记录命令、版本、状态码、数量和结果，不记录环境秘密。

## 8. 验证方式

### 8.1 静态验证

- `git diff --check`
- 目标 Markdown 和配置文件尾随空白扫描。
- 迁移、元数据、路由、API、脚本入口反向检查。
- 构建产物秘密和无关文件扫描。

### 8.2 应用验证

- `pnpm --filter @network-safe/server exec tsc -p tsconfig.json --noEmit`
- `pnpm --filter @network-safe/web exec vue-tsc -p tsconfig.json --noEmit`
- 服务端、前端、共享包和测试基础设施测试。
- `pnpm test:guided`
- `pnpm test:smoke`
- 代表性 `pnpm test:e2e`

### 8.3 发布验证

- `pnpm build:web`
- `pnpm build:server`
- 构建后的 Node 服务启动。
- nginx 静态页面、history fallback 和 API 代理。
- 全新或清理后的本机数据库初始化与幂等复跑。
