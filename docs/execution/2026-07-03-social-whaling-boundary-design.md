# 捕鲸攻击边界设计执行文档

## 1. 目标

本轮目标是为后续 `social/whaling` 捕鲸攻击实验锁定安全边界、案例化方向和分阶段落地顺序。

捕鲸攻击比普通网络钓鱼和鱼叉式钓鱼更强调高层角色、重大决策、授权例外、财务付款、法务合规和董事会 / 外部顾问语境带来的误判风险。本项目只允许通过固定虚构案例和防御流程复盘学习这些风险，不实现真实高管画像采集、真实组织结构收集、真实投递、凭据收集、模板生成、第三方平台调用或攻击脚本。

本轮只完成边界设计：

- 新增本文档。
- 同步 `docs/design/next-wave-security-labs.md`。
- 同步 `docs/TODO.md`。
- 同步 `docs/execution/security-lab-master-goal.md`。

本轮不创建 `labs/social/whaling/`、`tools/lab-scripts/social/whaling/`、页面、API、数据库字段、元数据或脚本。

## 2. 范围

### 2.1 本轮包含

- 明确 `social/whaling` 推荐模式为 `case-study`。
- 明确未来只能使用固定虚构高层决策案例、固定线索卡、固定风险标签和固定防御流程复盘。
- 明确攻击方视角只用于理解高权威请求为什么会降低核验意愿，不用于生成真实投递内容。
- 明确后续切片顺序和 ready 判定前置条件。
- 明确日志摘要只能记录固定 key、风险类别、决策和学习信号，不保存真实人员、邮箱、组织结构、职位链路、付款信息、链接、附件或凭据。

### 2.2 本轮不包含

- 不建立实验目录或 `meta.json`。
- 不新增前端路由、页面、API client 或页面模型。
- 不新增后端 service、接口或测试。
- 不新增脚本目录、`verify.ts` 或 `exploit.py`。
- 不接入第三方邮件、短信、社交平台、企业通讯录、CRM、HR、IM、会议系统、日程系统、收件箱或支付系统。
- 不采集真实高管姓名、邮箱、手机号、公司、部门、职位、汇报关系、董事会成员、外部顾问、供应商、客户或公开资料。
- 不生成可投递邮件、短信、IM 话术、会议邀请、标题库、正文模板库、附件诱导文案、跟踪链接、付款指令或群发脚本。

## 3. 现有依据

当前可复用样板包括：

- `social/phishing`：通用网络钓鱼识别训练，已按 `case-study ready` 收口。
- `social/spear-phishing`：鱼叉式钓鱼固定案例工作台，已按 `case-study ready` 收口。

`social/whaling` 后续必须继承这些边界。差异只在学习重点：

- 网络钓鱼重点观察相似域名、紧急语气、凭据请求、附件诱导和品牌仿冒。
- 鱼叉式钓鱼重点观察针对性上下文、角色权威、审批链绕过、供应商 / 人事 / 工程协作语境和二次确认缺失。
- 捕鲸攻击重点观察高层授权压力、重大业务事件、董事会或外部顾问语境、保密理由、时间窗口压缩和越级审批如何造成流程失守。

## 4. 推荐案例边界

后续可考虑以下固定虚构案例方向。它们只是案例方向，不是本轮接口字段，也不是可投递模板：

| 案例方向 | 学习重点 | 防御重点 |
|---|---|---|
| `executive-wire-approval` | 高层口吻与紧急大额付款造成审批压力 | 大额付款双人复核、可信通道回拨、付款冻结和审计留痕 |
| `board-confidential-request` | 董事会保密事项让普通流程被绕过 | 董事会事务固定通道、法务核验、最小知情范围和例外审批记录 |
| `legal-settlement-transfer` | 法务和外部顾问语境制造紧急合规压力 | 外部顾问白名单、案件编号核验、财务审批链和合规复核 |
| `ma-data-room-access` | 并购或投融资事件中的资料室访问诱导 | 数据室访问授权、临时权限有效期、独立审批和安全团队复核 |

这些案例必须满足：

- 使用虚构角色标签，不使用真实姓名、真实公司、真实部门、真实邮箱、真实手机号、真实职位链路或真实董事会关系。
- 只展示线索卡、风险标签、流程节点、决策后果和复盘问题。
- 不展示完整邮件正文、完整 IM 对话、可复制标题、可投递会议邀请、可投递附件名、真实链接或付款指令。
- 不允许用户输入真实组织信息、真实高管信息、真实顾问信息、真实付款信息或真实业务材料。

## 5. 攻击方视角学习模型

捕鲸攻击的攻击方视角应回答“为什么高权威和高保密语境会让人跳过核验”，但只能以抽象机制和固定案例观察呈现。

允许展示：

- 攻击者可能利用的误判机制：权威服从、保密压力、紧急时限、重大交易语境、越级请求、流程例外和职责边界不清。
- 固定案例中的风险线索：大额付款、外部顾问、董事会保密事项、临时资料室权限、审批链跳过和二次确认缺失。
- 攻击成功信号的学习化表达：错误放行、越级审批、未走可信通道、未记录例外审批、未触发安全复核。
- 修复版如何通过付款冻结、可信回拨、法务复核、董事会固定通道、最小授权和日志复盘改变决策。

禁止展示：

- 真实高管画像采集方法。
- 真实组织结构、汇报链或董事会关系挖掘方法。
- 可复制投递话术。
- 可投递邮件、短信、IM 或会议邀请模板。
- 可用于绕过安全网关、反垃圾系统、风控系统、审批系统或支付系统的技巧。
- 任何能直接面向第三方目标运行的脚本或流程。

## 6. 防御方视角学习模型

修复版后续应重点展示高风险流程防御，而不是只做单个字段校验。

建议覆盖：

- 高层请求核验：重要请求不能只凭显示名、职位、语气、保密要求或紧急程度判断。
- 可信通道确认：付款、权限、资料室、法务事项和例外审批必须通过预先约定的可信通道确认。
- 审批链复核：大额付款、供应商信息、法务和并购事项必须绑定正式审批链。
- 最小授权：临时资料访问必须有范围、有效期、审批记录和自动回收。
- 例外冻结：高风险例外先冻结，再通过财务、法务、安全或管理层复核。
- 隔离与举报：可疑链接、附件、会议邀请或请求先隔离，再通过安全流程复盘。
- 日志复盘：事件日志只展示风险类别、决策、学习信号和建议动作，不展示真实业务材料。

## 7. 后续切片建议

后续若继续实现，应按以下顺序推进，每一步都单独写执行文档：

1. 目录与 `planned` 元数据切片：建立 `labs/social/whaling/` 和基础文档，只登记 docs 入口。
2. 固定案例文档切片：补齐 `README.md`、`vuln/README.md`、`fixed/README.md`、`mock/README.md`、攻击方观察、修复说明和手动验证。
3. 后端固定案例 API 切片：只处理固定案例 key 和固定核验策略 key，接入统一事件日志安全摘要。
4. 前端固定案例工作台切片：只提供固定案例选择器、固定核验策略和复盘问题，不提供任意正文、邮箱、链接、附件、付款信息或人员信息输入。
5. 自动化验证切片：优先 API 差异测试、页面差异测试或只读一致性验证，不提供攻击脚本自动化。
6. case-study ready 收口切片：按主 goal 的 `case-study ready` 例外标准逐项审计。

## 8. 实施建议

- 优先复用 `social/phishing` 与 `social/spear-phishing` 的案例化结构、固定 key 模式、日志摘要方式和脚本边界。
- 后续实现前必须先通过类型定义、元数据规范和现有样板确认字段，不得猜测字段名。
- 页面应像决策复盘向导一样展示“观察高权威线索 -> 判断流程风险 -> 选择核验动作 -> 查看日志摘要 -> 完成复盘问题”。
- 后端未知固定 key 必须返回边界阻断结果，并且不得回显未知原始输入。
- 事件日志摘要只允许保存固定案例 key、风险标签数量、建议动作、学习信号和决策。
- `variants[].supportsAutomation` 不得因为存在页面测试、API 测试或只读验证脚本而误标为攻击脚本自动化。

## 9. 潜在风险分析

| 风险 | 影响 | 控制方式 |
|---|---|---|
| 案例过于真实，接近可投递内容 | 可能被误用为高管诈骗模板 | 只用线索卡、风险标签和流程节点，不写完整正文或话术 |
| 引入真实高管画像采集 | 可能收集隐私、组织结构或敏感业务关系 | 不提供 OSINT、通讯录、社交平台、CRM、HR、IM、会议或日程连接能力 |
| 日志保存真实业务材料 | 可能泄露敏感付款、法务或交易信息 | 只保存固定 key 和安全摘要，不保存姓名、邮箱、付款信息、链接、附件或原始文本 |
| 脚本边界不清 | 可能变成攻击工具 | 后续脚本只允许只读一致性验证，不提供 `exploit.py` |
| 防御流程过浅 | 学习者只记住“高层请求可疑”，不理解流程控制 | 强制覆盖大额付款复核、可信通道、法务核验、例外冻结和最小授权 |

## 10. 优化方案

- 后续可将 `social/phishing`、`social/spear-phishing` 与 `social/whaling` 的公共复盘组件复用，但不得提前抽象不稳定字段。
- 固定案例可按“财务 / 法务 / 董事会 / 并购资料室”四类组织，方便学习者比较不同高风险业务链路的防御点。
- 复盘问题应聚焦流程判断，例如“是否触发大额付款复核”“是否通过可信通道确认”“是否记录例外审批”“日志是否只保存安全摘要”。
- ready 收口前可增加只读一致性验证脚本，检查不存在 `exploit.py`、真实投递、真实画像采集、模板生成和第三方平台调用。
