## 1. 目标

建立一份可长期维护的安全覆盖矩阵，准确记录当前 65 个实验：

- 所属安全领域和场景 ID。
- 当前模式、实现方式和深度等级。
- 漏洞版/修复版或案例化双视角。
- 可参考的 OWASP、CWE、MITRE ATT&CK、供应链、云或 AI 安全体系。
- 当前自动化证据和长期强化动作。
- 当前平台完全缺失或明显偏薄的安全领域。

矩阵用于学习导航、范围审计和任务排序，不作为合规认证或真实攻击能力证明。

## 2. 范围

- 读取 65 个真实 `meta.json`。
- 读取共享元数据 schema 和 38 个引导式场景目录。
- 区分 27 个专用实现和 38 个通用引导式实现。
- 建立深度等级 D2/D3 的当前基线。
- 建立常见安全知识体系的导航映射。
- 列出结构性缺口和优先补强方向。
- 增加只读覆盖矩阵验证器和测试。
- 不修改 `meta.json` schema，不新增未经确认的字段。

## 3. 操作步骤

1. 从真实元数据统计实验数量、分类、模式、难度和风险。
2. 从 `guidedScenarioCatalog` 确认引导式场景 ID。
3. 为每个实验记录当前实现方式和深度等级。
4. 按场景根因映射常见 OWASP/CWE/ATT&CK 等参考体系。
5. 对不适合精确编号的高风险案例只记录体系和战术层级，不猜测编号。
6. 为每个场景记录长期动作：保持、专用化、扩展案例或升级工作台。
7. 汇总完全缺失和明显偏薄的领域。
8. 增加矩阵更新、审计和完成判定规则。
9. 反向检查矩阵是否包含全部 65 个元数据 ID。

## 4. 实施建议

- 深度等级和实施状态分开记录，`ready` 不等于最高深度。
- 标准映射优先使用稳定的漏洞类别或战术名称。
- 精确编号只在有较高把握时记录，并保留后续复核入口。
- 一个场景可以映射多个体系，但不为了表格完整强行填满。
- 结构性空白单独记录，不伪装成已有场景覆盖。
- 后续是否将映射写入 `meta.json`，必须另写 schema 设计文档。

## 5. 潜在风险分析

| 风险 | 影响 | 控制方式 |
|---|---|---|
| 标准编号映射错误 | 误导学习者 | 不确定时使用类别名称，不猜测编号 |
| `ready` 与深度混淆 | 错误判断项目成熟度 | 独立记录状态、模式和深度 |
| 65 个场景遗漏 | 矩阵不能作为审计基线 | 从真实元数据反向比对 65/65 |
| 标准版本变化 | 矩阵逐渐过时 | 记录更新时间并定期复核 |
| 为补矩阵修改 schema | 引入跨模块风险 | 本任务只写设计文档，不改字段 |

## 6. 优化方案

- 后续可增加只读脚本，从元数据生成数量、模式和深度统计。
- 后续可将标准映射维护为独立确认数据源，再生成 Markdown。
- 新实验进入 `ready` 前，将矩阵同步加入完成条件。
- 每次专用化升级后，更新深度、证据和长期动作。
