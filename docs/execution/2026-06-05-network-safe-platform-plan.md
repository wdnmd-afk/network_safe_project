# 网络安全学习平台执行文档

## 目标

在 Windows 本机环境中规划并建设一个基于 `Node.js + MySQL + Vue` 的网络安全学习平台，优先服务个人学习场景，并通过“漏洞版 / 修复版”对照方式承载常见网络安全问题的学习、验证与后续自动化测试。

## 范围

本阶段仅完成项目级文档与规则落地，不进入代码实现。

当前文档覆盖以下内容：

- 项目定位与建设目标
- 总体架构与模块边界
- monorepo 目录规划
- 场景组织规范
- 实验脚本目录规范
- 安全内容覆盖范围
- 后续建设阶段划分
- 风险边界与验证思路
- 项目级协作规则

## 操作步骤

### 1. 明确项目目标与约束

- 项目以个人学习优先。
- 每个重点场景同时提供漏洞版与修复版。
- 项目运行于 Windows 本机。
- 不使用 Docker。
- MySQL 使用本机 `localhost`。
- 前端允许打包交由 `nginx` 托管，后端 Node 服务独立运行。
- 仓库采用 monorepo 结构。

### 2. 定义平台总体架构

- 前端使用 Vue 构建统一学习入口。
- 后端使用 Node 构建平台核心服务。
- MySQL 存储平台数据、实验元数据、验证记录。
- 漏洞与实验内容按模块拆分为：
  - Web 漏洞靶场
  - 认证授权与业务逻辑靶场
  - 网络与基础设施实验模块
  - 社会工程学与新型攻击学习模块

### 3. 定义统一场景组织方式

- 每个场景尽量具备：
  - 场景说明
  - 攻击目标
  - 漏洞版入口
  - 修复版入口
  - 攻击步骤
  - 修复说明
  - 验证方式
  - 学习记录
- 所有场景通过统一元数据注册与加载。

### 4. 定义脚本与样本资源目录

- 单独设置实验脚本目录，不与前后端业务代码混放。
- 目录以场景为主组织，语言为辅。
- 主脚本语言为 Python 与 TypeScript。

### 5. 输出项目级文档

- 根目录 `README.md`
  - 描述项目定位、架构、目录、建设阶段与使用边界
- 根目录 `AGENTS.md`
  - 约束后续实现、文档、字段确认、测试与脚本边界
- `docs/design/`
  - 补充项目范围、安全内容清单与阶段规划
- `docs/TODO.md`
  - 持续记录工作进度

## 实施建议

### 建议 1：先做平台骨架与场景规范，不要先堆漏洞数量

如果没有统一的场景元数据、目录规范与验证模型，后续场景会迅速失控。优先把平台作为学习系统来设计，而不是做成一组零散 demo。

### 建议 2：对攻击类型采用分层覆盖

并非所有安全问题都适合做成真实 Web 靶场，应按以下方式分层：

- 可真实交互复现：优先建设
- 可脚本实验或本机模拟：受控纳入
- 更适合案例化演示：放入知识与演练模块

### 建议 3：平台核心与漏洞实现分层

- 平台核心负责用户、记录、题库、索引、日志、测试结果
- 漏洞场景负责实验行为本身
- 避免将所有漏洞逻辑直接揉进一个巨大的后端服务

### 建议 4：自动化测试从设计时就预留接入点

后续自动化测试不应临时追加，而应从场景元数据、脚本目录、验证结果结构开始预埋接口与规范。

## 潜在风险分析

### 风险 1：范围过大

用户给出的攻击全景图已经接近完整训练平台，而不是普通业务项目。若不做阶段拆分，极易导致：

- 文档空泛
- 实现周期过长
- 场景质量失衡
- 自动化测试难以收敛

### 风险 2：将所有攻击类型都强行做成同一种形态

不同攻击类型的演示方式天然不同。如果强行要求全部做成前端可直接点选复现的模式，会造成错误抽象和低质量实现。

### 风险 3：脚本目录失控

如果没有明确边界，实验脚本目录很容易演变为无规范的工具堆放区。必须从开始就限制其定位为本项目内的受控实验与验证资源。

### 风险 4：漏洞版与修复版耦合不清

如果实现层面没有分离 `vuln/` 与 `fixed/`，后续验证与回归会很混乱，也不利于学习对照。

### 风险 5：误用最新技术

尽量使用最新技术不等于盲目追预发布版本。平台底座应以稳定 LTS 与稳定大版本为主。

## 优化方案

### 优化方向 1：按阶段建设

建议最少拆成以下阶段：

1. 平台骨架与文档规范
2. Web 漏洞与认证授权一期场景
3. 网络与基础设施实验模块
4. 社会工程学与 AI 类学习模块
5. 自动化测试体系完善

### 优化方向 2：统一元数据驱动

后续页面展示、学习路径、记录、验证、自动化测试都尽量由场景元数据驱动，减少硬编码页面差异。

### 优化方向 3：脚本与平台共用场景标识

所有脚本目录、文档目录、测试目录、场景目录都围绕统一 `scene id` 组织，避免命名漂移。
