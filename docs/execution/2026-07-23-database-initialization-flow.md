## 1. 目标

将原有数据库占位说明和分散命令收口为可重复执行的 Windows 本机初始化流程，支持：

- 创建不存在的 MySQL 数据库。
- 按顺序应用仓库 SQL 迁移。
- 记录已执行迁移并支持幂等复跑。
- 对已有数据库进行显式、可审计的迁移基线登记。
- 补齐复盘表、认证种子和 65 个实验元数据种子。
- 避免数据库密码进入命令行、日志或 Git 文件。

## 2. 范围

- 新增 `database/scripts/apply-migrations.mjs`。
- 更新根 `package.json` 的 `db:migrate` 和 `db:prepare`。
- 更新 `database/README.md`。
- 增加数据库迁移脚本测试。
- 不修改现有业务表字段和现有迁移 SQL 语义。
- 不删除现有个人学习数据。

## 3. 操作步骤

1. 从 `apps/server/.env` 读取已确认的 `DATABASE_URL`。
2. 使用 Node 标准库解析 MySQL 主机、端口、用户、密码和数据库名。
3. 通过 `mysql.exe` 创建不存在的数据库。
4. 创建 `network_safe_schema_migrations` 迁移记录表。
5. 按文件名字典序应用 `database/migrations/*.sql`。
6. 成功执行迁移后写入迁移记录。
7. 对已记录迁移检查关键表仍然存在。
8. 通过显式 `--baseline-existing` 为结构完整的既有数据库登记迁移基线。
9. 通过 `db:prepare` 串联迁移、schema ensure、认证种子和实验种子。
10. 使用临时空库执行全流程后删除该临时库。

## 4. 实施建议

- 新迁移继续使用 `YYYYMMDD_description.sql` 命名。
- 迁移失败时不写成功记录，先处理 MySQL 错误再重试。
- `--baseline-existing` 只用于人工确认结构完整的已有数据库。
- MySQL 客户端固定使用 `utf8mb4`，避免中文种子写入失败。
- 密码通过子进程环境传递，不进入命令行参数。
- 新增或修改数据库表前继续先写独立执行文档。

## 5. 潜在风险分析

| 风险 | 影响 | 控制方式 |
|---|---|---|
| 已有库没有迁移记录 | 重复执行 `CREATE TABLE` | 只在显式参数下执行基线登记 |
| 迁移中途失败 | 数据库出现部分 DDL | 不写成功记录，保留错误并要求恢复后重试 |
| MySQL CLI 默认字符集错误 | 中文种子迁移失败 | 强制 `--default-character-set=utf8mb4` |
| 密码进入进程参数 | 本机信息泄露 | 使用子进程 `MYSQL_PWD`，并对错误输出脱敏 |
| 迁移记录存在但表被删除 | 产生假完成状态 | 每次复跑检查迁移关键表 |
