# 鱼叉式钓鱼边界设计执行文档

## 1. 目标

本轮目标是为后续 `social/spear-phishing` 实验锁定安全边界、案例化方向和分阶段落地顺序。

鱼叉式钓鱼比通用网络钓鱼更强调针对性上下文、角色压力、审批链和熟悉业务语境带来的误判风险。本项目只允许通过固定案例卡和防御复盘学习这些风险，不实现真实画像采集、真实投递、凭据收集、模板生成或攻击脚本。

本轮只完成边界设计：

- 新增本文档。
- 同步 `docs/design/next-wave-security-labs.md`。
- 同步 `docs/TODO.md`。
- 同步 `docs/execution/security-lab-master-goal.md`。

本轮不创建 `labs/social/spear-phishing/`、`tools/lab-scripts/social/spear-phishing/`、页面、API、数据库字段、元数据或脚本。

## 2. 范围

### 2.1 本轮包含

- 明确 `social/spear-phishing` 推荐模式为 `case-study`。
- 明确未来只能使用固定虚构案例、固定线索卡和固定防御检查清单。
- 明确攻击方视角只用于理解误判机制，不用于生成真实投递内容。
- 明确后续切片顺序和 ready 判定前置条件。
- 明确日志摘要只能记录固定 key、风险类别、决策和学习信号，不保存真实人员、邮箱、组织、链接、附件或凭据。

### 2.2 本轮不包含

- 不建立实验目录或 `meta.json`。
- 不新增前端路由、页面、API client 或页面模型。
- 不新增后端 service、接口或测试。
- 不新增脚本目录、`verify.ts` 或 `exploit.py`。
- 不接入第三方邮件、短信、社交平台、CRM、HR、IM、企业通讯录或收件箱服务。
- 不采集真实姓名、邮箱、手机号、公司、部门、职位、社交账号、组织关系或公开资料。
- 不生成可投递邮件、短信、IM 话术、标题库、正文模板库、附件诱导文案、跟踪链接或群发脚本。

## 3. 现有依据

当前可复用样板是 `social/phishing`：

- `mode: "case-study"`。
- `status: "ready"`。
- 页面和 API 只读取固定 `caseKey`、`reviewModeKey` 和 `defenseChecklistKey`。
- 只读脚本 `tools/lab-scripts/social/phishing/verify.ts` 只读取仓库文件，不发起 HTTP 请求。
- 已明确禁止真实邮件、短信、消息或链接投递，禁止真实凭据收集，禁止模板包和攻击脚本。

`social/spear-phishing` 后续必须继承这些边界。差异只在学习重点：

- 通用钓鱼重点观察相似域名、紧急语气、凭据请求、附件诱导和品牌仿冒。
- 鱼叉式钓鱼重点观察针对性上下文、角色权威、审批链绕过、供应商 / 人事 / 工程协作语境和二次确认缺失。

## 4. 推荐案例边界

后续可考虑以下固定虚构案例方向。它们只是案例方向，不是本轮接口字段，也不是可投递模板：

| 案例方向 | 学习重点 | 防御重点 |
|---|---|---|
| `executive-invoice-approval` | 角色权威与紧急付款审批压力 | 财务审批链、双人复核、回拨确认和付款变更冻结 |
| `vendor-payment-change` | 供应商付款信息变更带来的业务熟悉感 | 供应商主数据变更流程、可信渠道确认和异常变更审计 |
| `engineering-access-request` | 工程协作语境下的临时访问请求 | 最小权限、工单绑定、权限有效期和安全团队复核 |
| `hr-benefit-personalized` | 人事福利语境中的个性化诱导 | HR 官方入口、附件隔离、员工举报和内部公告核验 |

这些案例必须满足：

- 使用虚构角色标签，不使用真实姓名、真实公司、真实部门、真实邮箱或真实手机号。
- 只展示线索卡、风险标签、流程节点和复盘问题。
- 不展示完整邮件正文、完整 IM 对话、可复制标题、可投递附件名或真实链接。
- 不允许用户输入真实组织信息、真实联系人信息或真实业务材料。

## 5. 攻击方视角学习模型

鱼叉式钓鱼的攻击方视角应回答“为什么针对性信息会降低警惕”，但只能以抽象机制和固定案例观察呈现。

允许展示：

- 攻击者可能利用的误判机制：权威压力、紧急时限、业务熟悉感、跨部门请求、流程例外和二次确认缺失。
- 固定案例中的风险线索：角色压力、审批绕过、付款信息变更、临时权限请求、附件隔离缺失。
- 攻击成功信号的学习化表达：错误放行、流程绕过、缺少二次确认、风险线索未被标注。
- 修复版如何通过审批链、可信通道、最小权限和举报流程改变决策。

禁止展示：

- 禁止展示真实目标画像采集方法。
- 真实组织结构挖掘方法。
- 可复制投递话术。
- 可投递邮件或短信模板。
- 可用于绕过安全网关、反垃圾系统或风控系统的技巧。
- 任何能直接面向第三方目标运行的脚本或流程。

## 6. 防御方视角学习模型

修复版后续应重点展示流程防御，而不是只做单个字段校验。

建议覆盖：

- 角色核验：重要请求不能只凭显示名、职位或语气判断。
- 二次确认：付款、权限、凭据、附件和紧急例外必须通过可信通道确认。
- 审批链：财务、人事、供应商和工程权限变更必须绑定正式流程。
- 最小授权：临时访问必须有范围、有效期和审批记录。
- 隔离与举报：可疑链接、附件或请求先隔离，再通过安全流程复盘。
- 日志复盘：事件日志只展示风险类别、决策、学习信号和建议动作，不展示真实业务材料。

## 7. 后续切片建议

后续若继续实现，应按以下顺序推进，每一步都单独写执行文档：

1. 目录与 `planned` 元数据切片：建立 `labs/social/spear-phishing/` 和基础文档，只登记 docs 入口。
2. 固定案例文档切片：补齐 `README.md`、`vuln/README.md`、`fixed/README.md`、`mock/README.md`、攻击方观察、修复说明和手动验证。
3. 后端固定案例 API 切片：只处理固定案例 key 和固定检查策略 key，接入统一事件日志安全摘要。
4. 前端固定案例工作台切片：只提供固定案例选择器和复盘问题，不提供任意正文、邮箱、链接、附件或人员信息输入。
5. 自动化验证切片：优先 API 差异测试、页面差异测试或只读一致性验证，不提供攻击脚本自动化。
6. case-study ready 收口切片：按主 goal 的 `case-study ready` 例外标准逐项审计。

## 8. 实施建议

- 优先复用 `social/phishing` 的案例化结构、固定 key 模式、日志摘要方式和脚本边界。
- 后续实现前必须先通过类型定义、元数据规范和现有样板确认字段，不得猜测字段名。
- 页面应像学习向导一样展示“观察线索 -> 判断风险 -> 选择防御动作 -> 查看日志摘要 -> 完成复盘问题”。
- 后端未知固定 key 必须返回边界阻断结果，并且不得回显未知原始输入。
- 事件日志摘要只允许保存固定案例 key、风险标签数量、建议动作、学习信号和决策。
- `variants[].supportsAutomation` 不得因为存在页面测试、API 测试或只读验证脚本而误标为攻击脚本自动化。

## 9. 潜在风险分析

| 风险 | 影响 | 控制方式 |
|---|---|---|
| 案例过于真实，接近可投递内容 | 可能被误用为社工模板 | 只用线索卡和风险标签，不写完整正文或话术 |
| 引入真实画像采集 | 可能收集隐私或组织信息 | 不提供 OSINT、通讯录、社交平台、CRM、HR 或 IM 连接能力 |
| 日志保存真实业务材料 | 可能泄露敏感信息 | 只保存固定 key 和安全摘要，不保存姓名、邮箱、链接、附件或原始文本 |
| 脚本边界不清 | 可能变成攻击工具 | 后续脚本只允许只读一致性验证，不提供 `exploit.py` |
| 防御流程过浅 | 学习者只记住线索，不理解流程控制 | 强制覆盖审批链、二次确认、隔离举报和最小授权 |

## 10. 优化方案

- 后续可将 `social/phishing` 与 `social/spear-phishing` 的公共复盘组件复用，但不得提前抽象不稳定字段。
- 固定案例可按“财务 / 人事 / 工程 / 供应商”四类组织，方便学习者比较不同业务链路的防御点。
- 复盘问题应聚焦流程判断，例如“是否绕过正式审批链”“是否需要可信通道二次确认”“日志是否只保存安全摘要”。
- ready 收口前可增加只读一致性验证脚本，检查不存在 `exploit.py`、真实投递、模板生成和第三方平台调用。
