# 注入类一期剩余清单规划执行文档

## 1. 目标

本轮目标是对注入类一期剩余清单进行规划收口，明确 `web/nosql-injection`、LDAP 注入、XPath 注入、CRLF 注入的落地顺序、安全边界和后续执行入口。

本轮不直接实现新的实验代码，重点输出：

- 剩余注入类实验的优先级。
- NoSQL 注入是否适合真实交互靶场。
- LDAP / XPath / CRLF 分别采用案例化、模拟化或真实交互的判断依据。
- 后续进入具体实验实现前需要遵守的执行顺序和验证要求。

## 2. 范围

本轮覆盖：

- `web/nosql-injection`
- `web/ldap-injection`
- `web/xpath-injection`
- `web/crlf-injection`

本轮不包含：

- 不新增前端页面。
- 不新增后端 API。
- 不新增数据库迁移。
- 不新增实验脚本。
- 不引入 MongoDB、LDAP 服务、外部 XML 查询服务或代理服务。
- 不访问外部目标。

## 4. 实施建议

### 4.1 NoSQL 注入

建议作为下一项优先实验推进，但不引入真实 MongoDB。

推荐落地方式：

- 模式：`interactive`
- 业务包装：文档检索 / 优惠券筛选 / 账号画像查询预览。
- 后端实现：内存虚拟文档集合和受控查询解释器。
- 漏洞版：展示把用户输入合并进查询条件后导致条件语义被改变的风险。
- 修复版：只接受固定字段、固定比较方式和字符串关键词，不允许客户端提交查询结构。
- 日志：只记录查询模式、关键词长度、风险类别、是否命中受控样例和学习信号，不保存完整输入结构。

关键边界：

- 不连接真实 MongoDB。
- 不引入新的数据库依赖。
- 不支持任意查询对象执行。
- 不生成通用 NoSQL payload 库。

### 4.2 CRLF 注入

建议排在 NoSQL 之后，优先做真实交互式预览，但必须使用虚拟响应头构造器。

推荐落地方式：

- 模式：`interactive`
- 业务包装：下载文件名 / 跳转提示 / 邮件头预览。
- 后端实现：虚拟 HTTP 响应头构造器。
- 漏洞版：展示未规范化换行输入会造成响应头结构被污染的学习信号。
- 修复版：拒绝 CR / LF 控制字符，或将用户输入限制为允许列表字符集。

关键边界：

- 不构造真实响应拆分攻击。
- 不进行缓存投毒、Cookie 注入或代理链路演示。
- 只返回虚拟头部预览和学习信号。

### 4.3 XPath 注入

建议先规划为 `simulation` 或轻量交互实验，不优先接入真实 XML 查询引擎。

推荐落地方式：

- 模式：`simulation`
- 业务包装：XML 产品目录 / 权限策略查询预览。
- 后端实现：虚拟 XML 数据集合和固定查询路径模拟器。
- 漏洞版：展示输入影响查询条件后导致结果范围扩大。
- 修复版：固定查询模板、字段允许列表和文本值转义。

关键边界：

- 不读取真实 XML 文件。
- 不执行任意 XPath 表达式。
- 不接入外部 XML 数据源。

### 4.4 LDAP 注入

建议一期先做 `case-study`，暂不做真实交互靶场。

原因：

- 项目当前没有 LDAP 服务。
- 为单个实验引入目录服务会增加本机运行复杂度。
- LDAP 过滤器语义对初学者较抽象，案例化更适合先讲清“身份目录查询条件被拼接”的风险。

推荐落地方式：

- 模式：`case-study`
- 内容：登录筛选、组织成员搜索、权限组查询三个案例。
- 交互方式：给出输入、服务端拼接前后对比、修复后过滤器构造说明。

关键边界：

- 不连接真实 LDAP 服务。
- 不提供可对外使用的 LDAP 查询脚本。
- 不保存真实账号、组织结构或目录凭据。

## 5. 潜在风险分析

- NoSQL 注入若直接使用真实数据库，会引入新依赖和运行环境复杂度；本轮建议用虚拟文档集合规避。
- CRLF 注入若用真实 HTTP 头行为复现，容易越过学习平台边界；必须限定为虚拟头部预览。
- XPath 注入若执行任意表达式，可能演变成通用表达式执行器；必须限定固定教学映射。
- LDAP 注入若接入真实目录服务，会带来凭据、用户目录和服务配置风险；一期不建议真实复现。
- 四类实验都不得暴露 `inputSummaryJson`、真实密码、真实 token、真实 Cookie、完整 payload 或外部目标信息。

## 6. 优化方案

- 先复用命令注入、SSTI、XXE 的工作台结构：样例按钮、请求摘要、后端判定、学习信号、下一步建议。
- 先复用统一事件日志写入口，不为每个实验新增日志表。
- 先复用通用实验详情页，不为规划态实验新增独立详情能力。
- 对低优先级内容先补齐设计文档和占位元数据，再决定是否进入真实实现。
- 对每个具体实验仍单独写实现执行文档，避免把四类实验混在一次实现中。

## 7. 验证方式

本轮是规划文档任务，验证方式为：

- 文档存在且路径符合 `docs/execution/` 与 `docs/design/` 约定。
- TODO 与总纲已同步下一步状态。
- 文档行尾空白检查通过。
- `git diff --check` 不出现新增空白错误。

后续进入具体实验实现时，仍需按单个实验补齐：

- 后端 service / API 测试。
- 前端 API / helper / 路由测试。
- 共享元数据测试。
- `verify.ts` 本机验证计划。
- `exploit.py` 语法检查。
- 安全关键词扫描。

## 8. 下一步建议

下一步建议为 `web/nosql-injection` 单独编写实现执行文档，先锁定：

- 业务包装。
- 请求字段。
- 固定受控样例。
- 漏洞版学习信号。
- 修复版阻断信号。
- 事件日志摘要字段。
- 不引入 MongoDB 的虚拟文档查询器边界。
