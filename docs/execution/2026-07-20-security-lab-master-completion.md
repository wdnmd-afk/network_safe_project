## 3. 统一实现设计

### 3.1 共享固定场景目录

新增共享固定场景目录，准确声明每个新增实验的：

- 实验 id、分类、slug、标题、模式、风险等级和难度
- 固定观察案例与固定防御策略
- 漏洞版和修复版决策、学习信号、风险标签和下一步
- 安全边界和禁止能力

共享目录是新增实验字段的唯一来源。前端和后端不得通过多个可疑字段兜底，不得按标题猜测场景行为。

### 3.2 后端受控场景服务

新增通用受控场景服务：

- 只接受目录中声明的 `scenarioKey` 和 `controlKey`。
- 未知 key 必须阻断，且不得回显原始输入。
- 漏洞版只展示固定失效路径，不连接外部目标，不执行系统命令，不读取真实文件或网络数据。
- 修复版展示固定校验、隔离、最小权限、可信通道或审计策略。
- 每次评估写入 `lab_event_logs`，只记录固定 key、决策、信号、风险类别和摘要。
- 控制台日志继续通过统一 `recordLabEvent` 服务输出。

### 3.3 前端引导式工作台

视觉主题：深色、安静、面向学习操作的安全实验控制台，重点突出当前主题、变体、固定案例、控制策略和判定差异。

内容计划：

1. 顶部显示实验名称、模式、风险和当前变体。
2. 主工作区选择固定案例与固定控制策略。
3. 结果区展示服务端决策、学习信号、风险标签和下一步。
4. 辅助区提供漏洞版 / 修复版切换、返回详情和安全边界。

交互设计：

- 变体切换保持同一固定案例，便于对比。
- 评估结果使用短暂进入动画和明确状态色，但遵守减少动态效果偏好。
- 固定选项使用分段选择或按钮，不提供真实目标、任意正文、凭据、token、Cookie、付款数据、外部 URL 或命令输入。

新增主题复用同一个引导式工作台组件，但每个主题必须从共享目录读取自己的准确字段、文案、信号和边界。

### 3.4 脚本与验证

- 每个新增实验提供独立 `verify.ts` 入口。
- `verify.ts` 只读取当前实验元数据、文档、共享目录、前后端入口和测试文件，输出稳定 JSON 报告。
- 所有只读验证共享底层验证器，避免复制安全检查逻辑。
- `case-study` 变体的 `supportsAutomation` 必须保持 `false`，只读验证不能被标记为攻击自动化。
- Web / 认证受控模拟脚本只能面向 `localhost` / `127.0.0.1`，并要求显式目标参数校验。

## 5. 实施建议

- 优先建立共享目录和通用工作台，再批量落地目录与文档，避免 38 套重复业务逻辑。
- 共享不等于模糊：每个主题仍必须有独立 key、固定案例、控制策略、信号、风险标签、安全边界、文档和验证入口。
- 新增 API 保持单一请求结构：`scenarioKey`、`controlKey`，不得增加自由文本或外部目标字段。
- 所有高风险主题都使用固定虚构数据或静态状态机，不执行真实攻击行为。
- 文档只描述本机受控观察步骤和生产防护原则，不提供可迁移到外部目标的攻击操作链。

## 6. 潜在风险分析

| 风险 | 影响 | 控制方式 |
|---|---|---|
| 批量生成导致文档空泛 | 无法达到学习目标 | 每个主题在共享目录中声明独立案例、根因、防御策略、信号和边界，生成后执行字段完整性测试 |
| 通用 API 掩盖字段错误 | 前后端行为不一致 | 请求体固定为精确类型，服务端从共享目录校验 key，不做猜测式兜底 |
| 高风险主题变成攻击工具 | 超出项目安全边界 | 只使用固定案例和本机状态模拟，不连接外部服务，不提供真实 payload、投递、扫描或执行器 |
| 事件日志泄露输入 | 违反日志规范 | 仅保存目录中确认的固定 key 和计算结果，不保存原始未知输入 |
| 38 个主题缺少独立证据 | 无法证明完成 | 每个主题提供独立元数据、文档和 `verify.ts`，测试循环逐个输出结果 |
| 当前未提交修改被覆盖 | 丢失用户工作 | 修改前逐文件检查 diff，只追加或兼容现有改动 |
| 文档状态再次落后 | 总 goal 无法审计 | 最终从元数据和任务矩阵生成状态统计，并对 TODO/主 goal 做一致性测试 |

## 7. 优化方案

- 将新增主题数据集中在共享目录，页面、服务、文档生成器和验证器共用同一来源。
- 用动态路由承载新增工作台，保留已有专用实验页面和路由优先级，不重写现有实验。
- 用循环型单元测试覆盖全部目录项，新增主题时自动进入注册、API、元数据和文档检查。
- 将全量浏览器测试控制在代表性类别路径，所有主题的关键差异由服务测试和只读验证逐项覆盖。
- 最终增加任务矩阵一致性测试，阻止“元数据 ready、TODO 仍进行中”的再次发生。

## 8. 验证方式

按风险和范围分层执行：

1. 共享目录与元数据测试。
2. 后端通用服务和 API 测试，逐项覆盖 38 个新增主题。
3. 前端 API、状态模型、路由和组件测试。
4. 每个新增主题的 `verify.ts` 只读一致性验证。
5. Playwright 通用工作台代表性类别差异验证。
6. 前后端 TypeScript 类型检查。
7. `git diff --check` 和目标文件尾随空白扫描。
8. 危险能力关键词扫描，确认没有外部投递、真实凭据读取、网段扫描、恶意样本、任意命令执行或第三方平台连接。

非用户明确要求不执行全量生产构建；最终验证优先使用受影响范围测试、类型检查、脚本验证和代表性 E2E。
