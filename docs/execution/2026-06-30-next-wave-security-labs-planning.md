# 下一波安全实验规划执行文档

## 1. 目标

本轮目标是在 Web、注入类、认证授权和学习复盘主干已推进后，为下一波非 Web/Auth 扩展实验建立可执行规划。

本轮重点回答：

1. 网络 / 传输层、社会工程学、AI、供应链、恶意软件、客户端和基础设施实验先做哪些。
2. 哪些实验可以做本机模拟或交互页面。
3. 哪些实验必须保持案例化，不能真实复现。
4. 后续进入单个实验实现前，需要遵守哪些安全边界、目录落点和验证要求。

本轮只输出规划文档，不创建新实验脚本、页面、API、数据库迁移或元数据。

## 2. 范围

本轮覆盖以下扩展方向：

- 网络 / 传输层：端口扫描、DNS 劫持 / 污染、中间人攻击、ARP 欺骗、DNS 隧道、DDoS、BGP 劫持。
- 社会工程学：网络钓鱼、鱼叉式钓鱼、短信钓鱼、商业邮件诈骗、水坑攻击、钓鱼 WiFi、物理入侵。
- AI / 新型攻击：Prompt 注入、AI 驱动攻击、Deepfake、对抗性 AI、加密劫持。
- 供应链：依赖混淆、恶意包注入、更新投毒、硬件供应链。
- 恶意软件：勒索软件、木马、蠕虫、间谍软件、键盘记录器、Rootkit。
- 客户端攻击：Formjacking、驾车式下载、恶意浏览器插件、恶意广告、浏览器 MITB。
- 基础设施：配置错误利用、云安全漏洞、IoT 攻击、容器逃逸、零日漏洞利用。

本轮不包含：

- 不访问外部目标。
- 不扫描真实网段。
- 不修改本机 DNS、hosts、代理、路由、防火墙或浏览器扩展配置。
- 不发送真实邮件、短信、钓鱼链接或凭据收集表单。
- 不生成、运行、下载、改造恶意软件样本。
- 不发布、安装或伪装真实依赖包。
- 不调用外部 AI 服务生成仿冒身份、Deepfake 或攻击内容。
- 不创建可复用到外部目标的通用攻击脚本。

## 4. 推荐首批顺序

下一波建议按以下顺序推进：

| 顺序 | 实验 | 推荐模式 | 推荐原因 |
|---|---|---|---|
| 1 | `network/port-scan` | `simulation` | 作为网络实验入口，适合建立本机脚本和虚拟目标资产模型，但首版不做真实 socket 扫描 |
| 2 | `network/dns-hijack` | `simulation` / `case-study` | 可用内存 DNS 解析表展示解析结果被篡改的风险，不修改本机 DNS |
| 3 | `ai/prompt-injection` | `interactive` | 学习价值高，可通过确定性提示词路由模拟器实现，不需要外部 AI 服务 |
| 4 | `social/phishing` | `case-study` / 仿真页面 | 适合做邮件识别、表单风险判断和复盘题，不发送真实邮件或收集凭据 |
| 5 | `supply-chain/dependency-confusion` | `simulation` / `case-study` | 可用本地 manifest 分析和伪包元数据演示，不发布或安装真实包 |
| 6 | `infrastructure/misconfiguration` | `simulation` | 可用静态配置片段分析展示暴露面，不修改真实服务配置 |

该顺序的意图是先建立“高风险主题如何安全模拟”的范式，再扩展到更重的案例化模块。

## 5. 分类实施建议

### 5.1 网络 / 传输层

推荐先做 `network/port-scan` 与 `network/dns-hijack`。

`network/port-scan` 首版建议：

- 使用虚拟主机资产表和虚拟端口状态。
- 漏洞版展示“暴露服务过多 / 管理端口可见”的风险。
- 修复版展示“最小暴露面 / 端口访问控制 / 服务指纹收敛”的效果。
- 可在后续增加 localhost-only 验证脚本，但必须固定目标为 `127.0.0.1` / `::1`，固定端口清单，并默认只访问本项目服务端口。

禁止：

- 不扫描局域网、网段或外部 IP。
- 不做并发压力扫描。
- 不做服务漏洞识别或利用。
- 不绕过防火墙或认证。

`network/dns-hijack` 首版建议：

- 使用内存 DNS 解析表和固定域名样例。
- 漏洞版展示解析表被污染后用户访问错误目标的学习信号。
- 修复版展示 DNSSEC、可信解析源、证书校验和异常解析审计的防御思路。

禁止：

- 不修改本机 DNS、hosts、浏览器代理或路由。
- 不请求真实外部 DNS。
- 不实现真实劫持、投毒或隧道通信。

### 5.2 社会工程学

推荐先做 `social/phishing`。

首版建议：

- 使用仿真收件箱和固定邮件样例。
- 学习者从攻击方视角观察诱导链路，再从防御方视角标注风险点。
- 页面可以包含“可疑链接预览”“域名相似度”“紧急语气”“凭据索取”等固定信号。
- 事件日志只记录判断结果、风险标签和学习信号。

禁止：

- 不发送真实邮件、短信或链接。
- 不收集真实用户名、密码、token、Cookie、手机号或邮箱。
- 不生成可投递的钓鱼模板包。
- 不连接第三方邮件、短信或社交平台。

### 5.3 AI / 新型攻击

推荐先做 `ai/prompt-injection`。

首版建议：

- 使用确定性提示词路由模拟器，不调用外部 AI。
- 漏洞版展示系统指令、用户输入和检索片段混合后导致输出越界的风险。
- 修复版展示指令分层、工具调用允许列表、检索内容隔离和输出策略校验。
- 事件日志只记录输入长度、风险类别、是否命中受控样例和学习信号。

禁止：

- 不调用外部大模型执行攻击提示词。
- 不生成真实钓鱼、恶意代码、绕过策略或仿冒身份内容。
- 不保存完整提示词中的敏感数据。

Deepfake、对抗性 AI 和 AI 驱动攻击首批保持 case-study，不进入真实生成或攻击复现。

### 5.4 供应链

推荐先做 `supply-chain/dependency-confusion`。

首版建议：

- 使用本地固定 `package.json` 片段和伪包元数据。
- 漏洞版展示私有包命名与公共包命名冲突带来的解析风险。
- 修复版展示私有 registry 固定、scope 约束、lockfile 审计和安装源校验。

禁止：

- 不发布 npm / pnpm / PyPI / Maven 包。
- 不安装未知远程依赖。
- 不构造真实投毒包。
- 不生成真实恶意生命周期脚本。

### 5.5 恶意软件

恶意软件方向首批不做可执行样本。

推荐落地方式：

- 行为时间线。
- 固定日志样本分析。
- 安全化模拟状态机。
- 防御策略选择题和复盘题。

禁止：

- 不创建可执行恶意样本。
- 不实现加密、持久化、自传播、键盘记录、截屏、凭据窃取或隐藏进程能力。
- 不下载真实样本。
- 不提供绕过杀软或沙箱的说明。

### 5.6 客户端攻击

推荐优先考虑 `client/formjacking`，但放在社会工程学和供应链之后。

首版建议：

- 使用仿真结账页和固定输入字段。
- 漏洞版展示第三方脚本篡改表单提交目标的风险信号。
- 修复版展示 CSP、SRI、脚本清单、表单提交目标校验和敏感字段隔离。

禁止：

- 不采集真实银行卡、身份证、密码、token 或 Cookie。
- 不向外部地址发送任何表单数据。
- 不安装真实浏览器插件。

### 5.7 基础设施

推荐先做 `infrastructure/misconfiguration`。

首版建议：

- 使用静态配置片段分析器。
- 漏洞版展示调试端口开放、默认凭据提示、目录索引开启、过宽 CORS 等风险信号。
- 修复版展示最小暴露面、默认关闭、认证要求、配置审计和环境隔离。

禁止：

- 不修改真实 nginx、MySQL、Node、系统服务或云账号配置。
- 不连接真实云服务。
- 不复现容器逃逸、零日利用或 IoT 设备攻击。

## 6. 潜在风险分析

- 网络实验容易从教学脚本变成扫描工具；首版必须以虚拟资产为主，任何脚本都要固定 localhost 和端口清单。
- 社会工程学容易变成真实投递能力；必须限定为仿真页面、固定样例和识别训练。
- Prompt 注入容易输出可滥用内容；必须使用确定性模拟器，不把外部 AI 当作攻击执行器。
- 供应链实验容易触碰真实包生态；必须使用本地固定 manifest 和伪包元数据。
- 恶意软件与客户端攻击容易越过安全边界；必须先做案例化和安全化状态机。
- 基础设施实验若修改真实配置，会影响用户本机环境；必须使用静态片段和虚拟配置。

## 7. 优化方案

- 复用现有实验详情页、复盘卡片和 `lab_event_logs`，避免为每类内容新增孤立机制。
- 对所有高风险 case-study 实验沿用 `case-study ready` 元数据规则。
- 对 simulation 实验先建立 `mock/` 固定样本，再考虑页面和 API。
- 对脚本实验先写 README 和安全边界，再决定是否提供 `verify.ts` 或 `exploit.py`。
- 对无法安全自动化的实验，优先提供 Playwright 页面差异验证和只读一致性验证脚本。

## 8. 验证方式

本轮是规划文档任务，最小验证为：

- 新增执行文档位于 `docs/execution/`。
- 新增设计文档位于 `docs/design/`。
- `docs/TODO.md` 与 `docs/execution/security-lab-master-goal.md` 同步本轮进展。
- 目标文档通过行尾空白检查。
- 目标文档通过 `git diff --check`，若出现既有 LF/CRLF 提示需单独说明。
- 安全关键词扫描只允许命中文档中的禁止性说明，不应出现真实外部目标、真实凭据或可执行攻击链。
