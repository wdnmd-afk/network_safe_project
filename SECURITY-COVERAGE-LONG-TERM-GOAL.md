# 网络安全学习平台长期目标与安全覆盖健全路线

> 文档状态：长期生效
>
> 初始版本：2026-07-23
>
> 当前基线：65 个安全学习实验全量 `ready`

## 1. 文档定位

本文档是 `network_safe_project` 在完成首批 65 个安全实验之后的长期建设总目标，用于统一回答以下问题：

1. 当前平台已经具备哪些功能和安全内容。
2. 哪些安全类型只是完成了固定案例，哪些已经具备专用交互实现。
3. 当前安全覆盖还缺少哪些结构性领域。
4. 后续应按什么顺序逐步实现，而不是继续无序增加实验数量。
5. 每个新增或强化切片需要满足什么安全边界、验证标准和完成条件。
6. 平台功能、实验内容、测试体系、数据库、部署和工程治理如何同步演进。

本文档不替代以下文件：

- `AGENTS.md`：项目协作、安全边界和开发规则。
- `docs/execution/security-lab-master-goal.md`：首批 65 个实验的已完成目标。
- `docs/TODO.md`：具体实施切片和近期进度记录。
- `docs/execution/`：每项复杂任务开始前必须编写的详细执行文档。

后续不得直接把本文档中的全部规划一次性实现。每次只选择一个边界清晰、可独立验证的切片，先写执行文档，再实施、验证和提交。

## 2. 长期总目标

将当前“65 个安全类型已有可运行学习闭环”的平台，逐步建设为一个具备以下能力的 Windows 本机网络安全学习系统：

- 安全知识覆盖有清晰边界，并能映射常见安全知识体系。
- 每个安全主题都能区分攻击视角、根因、风险信号、修复策略和正常业务流程。
- 高价值主题优先具备专用漏洞版和修复版交互实现。
- 高风险主题通过固定样例、状态模拟、证据分析和决策复盘实现，不发展真实攻击能力。
- 学习者能够按路径学习、运行实验、记录结果、完成复盘并查看长期进度。
- 平台能够通过单元测试、API 测试、场景验证和端到端测试持续证明行为没有回归。
- 仓库能够在全新的 Windows 本机环境中按文档完成数据库初始化、构建、启动和验收。
- 文档、元数据、页面、API、脚本、数据库和测试状态始终保持一致。

## 3. 长期范围与非目标

### 3.1 长期建设范围

- 当前 65 个实验的持续维护和深度升级。
- Web、注入、认证授权、API 和业务逻辑安全。
- 网络、社会工程学、恶意软件、客户端和供应链安全。
- AI、云、基础设施、Windows 主机和身份基础设施安全。
- 密码学、秘密管理、检测响应和安全运营学习能力。
- 学习路径、复盘、题目、统计、搜索和内容管理能力。
- 自动化测试、覆盖矩阵、发布验收和本机部署能力。

### 3.2 明确不做

- 不连接、扫描、攻击或投递到未经授权的真实目标。
- 不创建通用扫描器、通用攻击框架或渗透接单工具箱。
- 不生成真实钓鱼邮件、短信、付款指令、凭据收集页或群发工具。
- 不创建、下载、编译、执行或传播真实恶意软件样本。
- 不读取真实浏览器凭据、Cookie、token、系统凭据或个人隐私数据。
- 不为了追求“数量齐全”而把所有主题都伪装成可真实攻击的交互实验。
- 不在没有执行文档、字段来源和验证证据的情况下标记完成。

## 5. 状态与深度模型

### 5.1 实施状态

继续沿用当前元数据状态：

- `planned`：已确认范围和边界，尚未形成可运行闭环。
- `in-progress`：已经进入实现，但尚未满足完成条件。
- `ready`：在本项目定义的模式和安全边界内，可运行、可观察、可验证。
- `archived`：保留历史记录，不再作为当前学习入口。

`ready` 只表示当前模式完成，不等于所有主题都达到同样的实操深度。

### 5.2 建议的深度等级

深度等级先记录在覆盖矩阵和设计文档中。是否写入 `meta.json`，必须另行设计并确认 schema 后再实施。

| 等级 | 名称 | 判定口径 |
|---|---|---|
| D0 | 清单级 | 只有名称、目标和边界说明 |
| D1 | 文档级 | 具备案例、攻击视角、修复说明和手工验证方案 |
| D2 | 引导式 | 具备固定场景、控制策略、页面、受控 API、事件日志和验证脚本 |
| D3 | 专用模拟 | 具备主题专用状态模型、页面/API 差异、正常流程和风险流程 |
| D4 | 专用交互 | 具备多步骤交互、状态变化、漏洞版/修复版闭环和自动化验证 |
| D5 | 完整学习闭环 | 在 D4 基础上具备学习路径、复盘题、统计、代表性 E2E 和发布回归证据 |

## 14. 单个实施切片标准流程

每个后续功能或实验按以下顺序推进：

1. 阅读现有实现、调用链、类型、数据库和相邻模块。
2. 编写独立执行文档，包含目标、范围、步骤、风险、优化和验证。

5. 确认安全边界和明确禁止能力。
6. 先设计共享类型、状态模型和数据库影响。
7. 实现漏洞视角、修复视角和正常业务流程。
8. 接入统一事件日志，只记录安全摘要。
9. 补齐标准目录、README、攻防文档和手工验证。
10. 补齐只读验证脚本和受影响范围测试。
11. 执行类型检查、静态检查和最小必要回归。
12. 更新 `docs/TODO.md`、覆盖矩阵和相关目标文档。
13. 按 Conventional Commits 分类提交。

## 15. 单个实验完成标准

新实验或深度升级实验进入 `ready` 前，至少满足：

- 有独立执行文档和明确的安全边界。
- 有准确、可验证的元数据。
- 有漏洞版和修复版，或满足高风险案例化例外。
- 有攻击视角、根因、风险信号、防御策略和正常流程说明。
- 页面与 API 只接受已确认字段和固定受控数据。
- 未知值被阻断且不回显原始敏感输入。
- 关键动作写入统一事件日志安全摘要。
- 有独立 README、攻击步骤、修复说明和手工验证。
- 有最小必要单元/API/前端测试和 `verify.ts`。
- 高价值交互主题有代表性页面级验证。
- 不包含外部目标连接、真实凭据读取、系统命令执行或恶意样本。
- 文档、元数据、路由、API、脚本和测试入口一致。

## 16. 实施建议

- 优先补“结构性空白”和“已经标为 interactive 但深度不足”的主题。
- 对社会工程、恶意软件、零日和容器逃逸等高风险主题，优先深化证据分析和防御决策，不深化真实攻击操作。
- API 和业务逻辑实验优先复用现有用户、商品、订单、学习记录和事件日志模型。
- 不重复创建同义主题，例如 API BOLA 应优先复用和扩展现有 `auth.idor`。
- 新分类应先确认共享目录、页面路由、种子、数据库和统计是否支持，再开始创建场景。
- 通用工作台升级应保持现有专用实验路由优先级，避免行为回归。
- 先建立覆盖矩阵，再决定是否修改 `meta.json` schema，避免为了文档统计贸然增加字段。
- 每轮控制在一个完整闭环，验证通过后再进入下一项。

## 17. 潜在风险分析

| 风险 | 影响 | 控制方式 |
|---|---|---|
| 继续只追求实验数量 | 名称越来越多但学习深度不提升 | 优先执行覆盖矩阵和深度等级审计 |
| `ready` 被误解为真实靶场完成 | 用户错误判断项目能力 | 区分实施状态和深度等级 |
| 通用工作台过度复用 | 不同主题只剩文案差异 | 建设多步骤第二版并专用化高价值主题 |
| 新增分类破坏现有统计和种子 | 元数据、数据库和页面不一致 | 先做全链路设计和一致性测试 |
| 高风险主题越过安全边界 | 项目变成攻击工具 | 只使用固定数据、静态样本和虚拟状态 |
| 日志保存原始输入 | 产生敏感数据风险 | 继续只记录固定 key 和计算后的安全摘要 |
| 数据库迁移不可重复 | 新环境无法启动 | 建立迁移顺序、状态检查和恢复说明 |
| 文档长期落后 | 无法判断真实进度 | 每个切片强制同步 TODO、矩阵和目标文档 |
| 大范围重构造成回归 | 65 个现有实验受影响 | 逐项切片、保持专用路由优先并执行回归 |
| CI 或构建暴露秘密 | 本机配置泄露 | 只使用示例值和秘密注入，不打印环境变量 |
| 长期目标无限膨胀 | 项目失去个人学习定位 | 每阶段按学习价值、安全性和维护成本重新排序 |

## 18. 优化方案

- 从共享目录生成覆盖矩阵的基础统计，减少手工漂移。
- 为通用工作台建立可组合的证据卡、时间线和分支模型。
- 为每个弱分类先建设一个高质量纵向样板，再扩展同类主题。
- 统一场景验证器，复用安全边界、路径和元数据检查。
- 将轻量检查与高成本检查分层：日常运行局部测试，发布阶段运行完整验收。
- 将平台部署、实验覆盖和内容深度分别统计，不再用单一百分比混合表示。
- 对旧实验采用“保持行为、补标准映射、补正常流程”的渐进式增强方式。

## 19. 验证方式

### 19.1 文档和静态一致性

- `git diff --check`
- 目标 Markdown 行尾空白检查。
- 65 个元数据、目录、文档和脚本入口反向审计。
- 覆盖矩阵与真实 `meta.json` 的数量、分类和模式统计对比。

### 19.2 默认轻量验证

- `pnpm test:shared`
- `pnpm test:guided`
- 受影响的服务端测试。
- 受影响的前端测试。
- 前后端 TypeScript 类型检查。
- 受影响场景的 `verify.ts`。

### 19.3 集成和页面验证

- `pnpm test:smoke`
- 受影响分类的代表性 Playwright。
- 新增 API、事件日志和数据库写入的集成测试。
- 新增路由和元数据 entrypoints 一致性测试。

### 19.4 发布验证

只有在用户明确授权后执行：

- `pnpm build:web`
- `pnpm build:server`
- 构建产物启动验证。
- nginx 静态托管和 `/api` 反向代理验证。
- 全新 Windows 本机数据库初始化和端到端验收。

## 20. 长期完成判定

本文档是持续演进目标，不以“所有网络安全知识全部完成”作为终点。满足以下条件时，可以认为平台进入成熟维护阶段：

- 当前主流应用安全领域不存在明显结构性空白。
- 每个正式安全分类至少有一个专用纵向样板。
- 现有 65 个实验全部具有标准映射和深度等级。
- 所有 `interactive` 实验都具有与其名称相符的交互深度。
- 高风险案例类实验具备多步骤证据分析和防御决策能力。
- API、业务逻辑、密码学、Windows 主机、云原生和检测响应均有代表性实验。
- 学习路径、进度、验证、复盘和统计形成完整闭环。
- 新环境安装、数据库初始化、构建、启动和验收可重复。
- 测试、CI、发布和文档同步机制稳定运行。
- 安全边界审计持续证明项目没有扩展成对外攻击工具。
